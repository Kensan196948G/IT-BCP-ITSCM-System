## 1. 概要

### 1.1 目的

本文書は、IT事業継続管理システムの災害復旧（DR: Disaster Recovery）テストの仕様を定義する。本システムは Azure Container Apps 上で東日本リージョン（プライマリ）と西日本リージョン（セカンダリ）のマルチリージョン構成で運用されており、災害時のフェイルオーバー、データ整合性、オフライン動作、RTO/RPOの達成を検証する。

### 1.2 DRテストの分類

| テスト種別 | 説明 | 頻度 |
|-----------|------|------|
| フェイルオーバーテスト | リージョン間のフェイルオーバー動作検証 | 月次（自動） + 四半期（手動） |
| オフライン動作テスト | PWAのオフラインモード動作検証 | 月次 |
| RTOトラッカーテスト | RTO/RPO計測と追跡の正確性検証 | 月次 |
| 通知テスト | 災害時通知の確実な配信検証 | 月次 |
| データ復旧テスト | バックアップからのデータ復旧検証 | 四半期 |
| 統合DRテスト | 全DR機能の統合シナリオテスト | 半期 |

### 1.3 RTO/RPO 目標

| 項目 | 目標値 | 備考 |
|------|--------|------|
| RTO（復旧目標時間） | 90秒以内 | フェイルオーバー完了まで |
| RPO（復旧目標地点） | 5分以内 | データ損失許容範囲 |
| MTTR（平均復旧時間） | 60秒以内 | フェイルオーバーの平均値 |

## 3. フェイルオーバーテスト

### 3.1 テスト仕様一覧

#### DR-FO-001: 東日本→西日本フェイルオーバー（計画的）

| 項目 | 内容 |
|------|------|
| テストID | DR-FO-001 |
| テスト名 | 計画的フェイルオーバー（East → West） |
| 目的 | 計画的なリージョン切替が90秒以内に完了することを検証 |
| 前提条件 | 両リージョンが正常稼働中。データレプリケーションが同期済み |
| 合格基準 | フェイルオーバー完了まで90秒以内。データ損失なし。エラー率0% |

**テスト手順:**

| ステップ | 操作 | 確認事項 | 計測項目 |
|---------|------|---------|---------|
| 1 | 東日本リージョンのヘルスチェック確認 | 全サービスが正常 | - |
| 2 | 西日本リージョンのヘルスチェック確認 | 全サービスが正常（スタンバイ） | - |
| 3 | PostgreSQLレプリケーションラグ確認 | ラグが5秒以内 | レプリケーションラグ |
| 4 | Azure Front Doorのルーティング変更（East→West） | 設定変更成功 | 設定変更所要時間 |
| 5 | 西日本PostgreSQLのプロモーション（Read Replica → Primary） | プロモーション成功 | プロモーション所要時間 |
| 6 | 西日本Container Appsのスケールアウト | レプリカ数が本番相当に | スケールアウト所要時間 |
| 7 | フロントエンドからのAPI呼び出し | レスポンスが正常 | API応答時間 |
| 8 | WebSocket接続の確認 | RTOダッシュボードが更新される | 接続確立時間 |
| 9 | 全機能の動作確認 | BCP計画閲覧、訓練データ表示等 | - |
| 10 | 合計フェイルオーバー時間計測 | 90秒以内 | 総フェイルオーバー時間 |

**期待される計測値:**

| 項目 | 目標値 | 許容範囲 |
|------|--------|---------|
| Front Door ルーティング切替 | 30秒以内 | 45秒まで |
| DB プロモーション | 30秒以内 | 45秒まで |
| アプリケーション起動確認 | 15秒以内 | 30秒まで |
| 総フェイルオーバー時間 | 60秒目標 | 90秒以内 |

| 5 | RTO残り時間の色変化確認 | 50%以下: 黄色、25%以下: 赤色 | - |
| 6 | 復旧完了を記録 | RTOカウントダウン停止 | - |
| 7 | 実績RTOの表示確認 | 実際の復旧時間が表示 | 実績RTO精度 |
