## 1. 概要

### 1.1 BIAの目的

ビジネスインパクト分析（BIA）は、ITサービスの中断が事業活動に与える影響を体系的に分析し、以下を明確化するための機能である。

1. **重要業務プロセスの特定**: 業務プロセスの重要度を評価し、優先的に保護すべきプロセスを特定する
2. **ITシステム依存関係の明確化**: 各業務プロセスが依存するITシステムを特定し、依存度を評価する
3. **RTO/RPOの根拠提供**: 業務影響度に基づいてRTO/RPOの設定根拠を提供する
4. **財務影響の定量化**: 業務中断による財務的影響を試算する
5. **規制違反リスクの評価**: 業務中断に伴う規制違反のリスクを評価する

### 1.2 BIAの対象範囲

- みらい建設のIT関連業務プロセス全般
- IT関連業務を支えるITシステム全般
- ITサービスに関連する社内外のステークホルダー

### 1.3 BIAの実施頻度

| 実施タイミング | 内容 |
|-------------|------|
| 定期実施 | 年1回（年度初め） |
| 臨時実施 | 重大な組織変更、システム変更、新規システム導入時 |
| 見直し | 四半期ごとの妥当性確認 |
| インシデント後 | 重大インシデント発生後の再評価 |

## 3. BIA-001: 業務プロセス影響評価

| 項目 | 内容 |
|------|------|
| 要件ID | BIA-001 |
| 要件名 | 業務プロセス影響評価 |
| 優先度 | 必須 |
| 概要 | 業務プロセスの中断がビジネスに与える影響を体系的に評価し、復旧優先度の根拠を提供する |

### 3.1 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| 業務プロセス名 | テキスト | 必須 | 評価対象の業務プロセス名 |
| プロセスID | 自動採番 | 自動 | プロセスの一意識別子 |
| 所管部門 | 選択 | 必須 | プロセスを所管する部門 |
| プロセス概要 | テキスト | 必須 | 業務プロセスの概要説明 |
| 依存ITシステム | 選択（複数可） | 必須 | プロセスが依存するITシステム |
| 依存度 | 選択 | 必須 | 各ITシステムへの依存度（完全依存/高依存/中依存/低依存/依存なし） |
| 業務影響度 | 選択 | 必須 | 中断時の業務影響度（5段階: 壊滅的/重大/中程度/軽微/無視可能） |
| 時間経過別影響 | テーブル | 必須 | 停止時間ごとの影響度変化（1h/4h/8h/24h/48h/72h/1w） |
| 影響を受けるステークホルダー | 選択（複数可） | 必須 | 社内部門/顧客/取引先/規制当局 |
| ピーク期間 | テキスト | 任意 | 業務の繁忙期（影響が大きくなる期間） |
| 手動代替手段 | テキスト | 任意 | IT停止時の手動代替手段 |
| 手動代替可能時間 | 時間 | 任意 | 手動代替で運用可能な最大時間 |
| 評価者 | ユーザ選択 | 必須 | 評価を実施した担当者 |
| 評価日 | 日付 | 必須 | 評価実施日 |

### 3.2 処理

1. **業務プロセス登録**: 全業務プロセスをカタログとして登録。階層構造（大分類→中分類→小分類）で管理。
2. **影響度評価**: 5段階の影響度評価マトリクスを使用。時間経過に伴う影響度の変化を曲線で記録。
3. **依存関係マッピング**: 業務プロセスとITシステムの依存関係を自動マッピング。依存度の定量化。
4. **RTO推奨値算出**: 影響度評価と依存関係から、各ITシステムのRTO推奨値を自動算出。算出ロジック: 最も影響度が高い業務プロセスの「許容可能な最大停止時間」を基準。
5. **整合性チェック**: 評価結果と現行RTO定義値の整合性を自動チェック。乖離がある場合はアラートを表示。
6. **レビュー管理**: 評価結果のレビュー・承認ワークフロー。

### 3.3 影響度評価基準

| 影響度レベル | スコア | 定義 | 業務影響の例 |
|------------|--------|------|------------|
| 壊滅的（Critical） | 5 | 事業の存続に関わる影響 | 全社的な業務停止、重大な契約違反、生命・安全への影響 |
| 重大（High） | 4 | 広範囲の業務に重大な支障 | 主要部門の業務停止、大口顧客への影響、大幅な売上損失 |
| 中程度（Medium） | 3 | 特定業務に顕著な支障 | 特定部門の業務遅延、一部顧客への影響、中程度の損失 |
| 軽微（Low） | 2 | 限定的な業務への軽微な影響 | 業務効率の低下、内部業務の遅延 |
| 無視可能（Negligible） | 1 | ほぼ影響なし | 一時的な不便、代替手段で即座に対応可能 |

### 3.4 時間経過別影響度評価テンプレート

| 停止時間 | 業務影響度 (1-5) | 財務影響 | 顧客影響 | 規制影響 | 評判影響 | 総合影響度 |
|---------|-----------------|---------|---------|---------|---------|----------|
| 1時間 | | | | | | |
| 4時間 | | | | | | |
| 8時間 | | | | | | |
| 24時間 | | | | | | |
| 48時間 | | | | | | |
| 72時間 | | | | | | |
| 1週間 | | | | | | |

### 3.5 出力

| 出力項目 | 形式 | 説明 |
|---------|------|------|
| 業務プロセスカタログ | テーブル表示 | 全業務プロセスの一覧 |
| 影響度評価マトリクス | ヒートマップ | プロセス × 時間の影響度マップ |
| 依存関係マップ | ネットワークグラフ | プロセス-システム間の依存関係 |
| RTO推奨値レポート | テーブル + グラフ | 各システムのRTO推奨値と根拠 |
| 整合性チェック結果 | テーブル表示 | 現行定義との乖離一覧 |
| BIA評価報告書 | PDF | 総合的なBIA評価レポート |

### 3.6 画面イメージ

1. **業務プロセスカタログ画面**: ツリー形式で業務プロセスを表示。各プロセスに影響度バッジ（色分け: 赤/橙/黄/緑/灰）。クリックで詳細表示。
2. **影響度評価入力画面**: 上部にプロセス基本情報。中央に時間経過別影響度の入力テーブル。右にリアルタイムの影響度曲線グラフ。下部にITシステム依存関係の設定。
3. **依存関係マップ画面**: インタラクティブなネットワークグラフ。業務プロセスノード（青）とITシステムノード（緑）を線で接続。線の太さで依存度を表現。

### 3.7 受け入れ基準

1. 業務プロセスの階層的な登録・管理が可能であること
2. 5段階の影響度評価が記録できること
3. 時間経過別の影響度変化が記録・グラフ表示されること
4. ITシステムとの依存関係がマッピングできること
5. RTO推奨値が自動算出されること
6. 現行RTO定義との整合性チェックが動作すること
7. レビュー・承認ワークフローが動作すること
8. BIA評価報告書のPDF出力が可能であること

## 4. BIA-002: 財務影響試算

| 項目 | 内容 |
|------|------|
| 要件ID | BIA-002 |
| 要件名 | 財務影響試算 |
| 優先度 | 必須 |
| 概要 | 業務中断による財務的影響（売上損失、追加コスト、違約金等）を定量的に試算する |

### 4.1 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| 対象業務プロセス | 選択 | 必須 | BIA-001で登録済みの業務プロセス |
| 売上影響 | 金額/時間 | 必須 | 1時間あたりの売上損失額 |
| 売上影響算出根拠 | テキスト | 必須 | 売上損失額の算出根拠 |
| 追加人件費 | 金額/時間 | 任意 | 手動代替運用による追加人件費 |
| 残業・緊急対応費 | 金額/時間 | 任意 | 復旧作業の残業・緊急対応費 |
| 外注費 | 金額 | 任意 | ベンダー緊急対応費 |
| 違約金・ペナルティ | 金額 | 任意 | 契約違反に基づく違約金 |
| 規制上の罰金 | 金額 | 任意 | 規制違反に基づく罰金 |
| 顧客離反損失 | 金額 | 任意 | 長期的な顧客離反による損失 |
| ブランド毀損損失 | 金額 | 任意 | ブランド・評判の毀損による損失（推定） |
| 保険カバー範囲 | 金額 | 任意 | サイバー保険等によるカバー額 |
| 通貨 | 選択 | 必須 | 日本円（JPY） |

### 4.2 処理

1. **影響額自動計算**: 停止時間に応じた累積影響額を自動計算。
   - 総影響額 = (売上影響 × 停止時間) + 追加人件費 + 外注費 + 違約金 + 罰金 + 顧客離反損失 + ブランド毀損損失 - 保険カバー
2. **時間経過別試算**: 1時間〜1週間の各時点での累積影響額をシミュレーション。
3. **シナリオ別試算**: 楽観的/標準/悲観的の3パターンで試算（係数: 楽観0.5倍、標準1.0倍、悲観2.0倍）。
4. **システム別集計**: 業務プロセスの影響をITシステム単位に集計。最も影響額の大きいシステムを特定。
5. **年間期待損失額（ALE）**: 発生確率 × 影響額 = 年間期待損失額の算出。
6. **投資対効果**: BCP投資額と期待損失低減額の比較分析（ROI）。

### 4.3 財務影響カテゴリ

| カテゴリ | 説明 | 算出方法 |
|---------|------|---------|
| 直接損失 | 売上損失、生産停止損失 | 1時間あたり売上 × 停止時間 |
| 追加コスト | 復旧費用、緊急対応費、外注費 | 実績ベース or 見積り |
| 契約上の損失 | 違約金、SLAペナルティ | 契約条件に基づく |
| 規制上の損失 | 行政罰、課徴金 | 規制条文に基づく |
| 間接損失 | 顧客離反、ブランド毀損、市場機会損失 | 推定値（過去事例ベース） |
| 保険回収 | サイバー保険等による補填 | 保険契約に基づく |

### 4.4 出力

| 出力項目 | 形式 | 説明 |
|---------|------|------|
| 時間経過別影響額グラフ | 折れ線グラフ | 停止時間ごとの累積影響額 |
| シナリオ別比較 | 棒グラフ | 楽観/標準/悲観の比較 |
| システム別影響額 | 棒グラフ（横） | システムごとの影響額ランキング |
| カテゴリ別内訳 | 円グラフ | 損失カテゴリの内訳 |
| 年間期待損失額 | テーブル | ALE算出結果 |
| ROI分析 | テーブル + グラフ | BCP投資の対効果分析 |
| 財務影響レポート | PDF | 総合的な財務影響レポート |

### 4.5 画面イメージ

1. **財務影響入力画面**: フォーム形式で各影響カテゴリの金額を入力。右側にリアルタイムでグラフが更新。自動計算フィールドが即座に反映。
2. **シミュレーション画面**: スライダーで停止時間を変更し、影響額の変化をリアルタイムで確認。シナリオ切替（楽観/標準/悲観）。
3. **影響額ダッシュボード**: 全業務プロセスの影響額を集約表示。システム別の影響額ランキング。

### 4.6 受け入れ基準

1. 各影響カテゴリの金額入力が可能であること
2. 停止時間に応じた累積影響額が自動計算されること
3. 3パターン（楽観/標準/悲観）のシナリオ試算が可能であること
4. システム別の影響額集計が正しく行われること
5. 年間期待損失額（ALE）が算出されること
6. 影響額グラフが正しく表示されること
7. PDF形式でのレポート出力が可能であること

## 5. BIA-003: 規制違反リスク評価

| 項目 | 内容 |
|------|------|
| 要件ID | BIA-003 |
| 要件名 | 規制違反リスク評価 |
| 優先度 | 必須 |
| 概要 | 業務中断に伴う規制違反のリスクを評価し、コンプライアンス上の対策を特定する |

### 5.1 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| 関連規制名 | テキスト | 必須 | 規制・法令の名称 |
| 規制カテゴリ | 選択 | 必須 | 法令/業界規制/契約上の義務/社内規程 |
| 規制条項 | テキスト | 必須 | 具体的な条項番号と内容 |
| 対象業務プロセス | 選択（複数可） | 必須 | 規制が適用される業務プロセス |

### 5.2 処理

1. **規制カタログ管理**: 関連する法規制・契約義務をカタログとして管理。規制の変更・追加を追跡。
2. **規制-プロセス-システムマッピング**: 規制 → 業務プロセス → ITシステムの三層マッピング。
3. **リスク評価**: 各規制違反のリスクを「発生可能性 × 影響度」で評価。リスクマトリクスでの視覚化。

5. **対策管理**: リスク低減策の登録・進捗管理。対策の有効性評価。
6. **変更監視**: 規制の変更を追跡し、影響分析を実施（手動トリガー）。

## 8. 評価方法論

### 8.1 定量評価手法

#### 8.1.1 RTO推奨値の算出ロジック

```
RTO推奨値 = MIN(
    業務プロセス影響に基づくRTO,
    規制上の許容停止時間,
    契約上のSLA要件
)

業務プロセス影響に基づくRTO:
  影響度曲線において「中程度（スコア3）」に到達する時間を基準とする。
  ただし、複数のプロセスが同一システムに依存する場合は、
  最も厳しい（最短の）RTOを採用する。
```

#### 8.1.2 年間期待損失額（ALE）の算出

```
ALE = SLE × ARO

SLE（Single Loss Expectancy: 1回あたりの損失額）
  = 直接損失 + 追加コスト + 契約上の損失 + 間接損失 - 保険回収

ARO（Annual Rate of Occurrence: 年間発生頻度）
  - 大規模地震: 0.03（約30年に1回）
  - ランサムウェア攻撃: 0.15（約7年に1回）
  - DC障害: 0.1（約10年に1回）
  - クラウド広域障害: 0.5（年に0.5回程度）
  - パンデミック: 0.05（約20年に1回）
  - サプライヤー障害: 0.2（約5年に1回）
  - 局所的システム障害: 2.0（年に2回程度）
```

### 8.2 定性評価基準

#### 8.2.1 影響度-発生可能性マトリクス

|  | 発生可能性: 極低 (1) | 発生可能性: 低 (2) | 発生可能性: 中 (3) | 発生可能性: 高 (4) | 発生可能性: 極高 (5) |
|------|:---:|:---:|:---:|:---:|:---:|
| 影響度: 壊滅的 (5) | 中 | 高 | 極高 | 極高 | 極高 |
| 影響度: 重大 (4) | 低 | 中 | 高 | 極高 | 極高 |
| 影響度: 中程度 (3) | 低 | 中 | 中 | 高 | 高 |
| 影響度: 軽微 (2) | 極低 | 低 | 低 | 中 | 中 |
| 影響度: 無視可能 (1) | 極低 | 極低 | 低 | 低 | 中 |

#### 8.2.2 リスクレベルの対応方針

| リスクレベル | 色 | 対応方針 |
|------------|-----|---------|
| 極高 | 赤 | 即座に対策実施。経営層の承認のもと優先的にリソースを配分 |

| 低 | 緑 | 次回BIAレビュー時に再評価。現状維持も許容 |
| 極低 | 灰 | 受容可能。モニタリングのみ |
