## 1. BCP/ITSCM計画管理（PLAN）

### PLAN-001: IT復旧計画文書管理

| 項目 | 内容 |
|------|------|
| 要件ID | PLAN-001 |
| 要件名 | IT復旧計画文書管理 |
| 優先度 | 必須 |
| 概要 | IT復旧計画（IT-DRP）文書の作成、レビュー、承認、版管理、配布を行う統合文書管理機能 |

#### 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| 計画名 | テキスト（最大200文字） | 必須 | IT復旧計画の名称 |
| 対象システム | 選択（複数可） | 必須 | 復旧計画の対象システム（マスタから選択） |
| 対象シナリオ | 選択（複数可） | 必須 | 対応する災害シナリオ（SCN-001〜007） |
| 計画本文 | リッチテキスト | 必須 | 復旧計画の本文（Markdownエディタ対応） |
| 添付ファイル | ファイル（PDF/Word/Excel/画像） | 任意 | 補足資料、手順書、構成図等 |
| 有効期限 | 日付 | 必須 | 計画の有効期限（レビュー期限） |
| 承認者 | ユーザ選択 | 必須 | 承認権限を持つユーザ |
| レビュワー | ユーザ選択（複数可） | 任意 | レビュー担当者 |
| タグ | テキスト（複数可） | 任意 | 分類用タグ |
| 関連計画 | 計画選択（複数可） | 任意 | 関連する他の復旧計画 |

#### 処理

1. **計画作成**: テンプレートベースの計画作成機能を提供する。ISO20000 ITSCMの要件に基づいたテンプレートを標準提供する。
2. **版管理**: 計画の変更履歴を自動記録する。メジャーバージョン（承認済み版）とマイナーバージョン（ドラフト版）を区別する。任意のバージョン間の差分比較を可能にする。
3. **レビュー・承認ワークフロー**:
   - ドラフト → レビュー依頼 → レビュー中 → レビュー完了 → 承認依頼 → 承認済み → 配布済み
   - レビューコメント機能（行単位のインラインコメント対応）
   - 承認者による電子承認（タイムスタンプ付き）
   - 差し戻し機能（理由記入必須）
4. **有効期限管理**: 有効期限到達の30日前、14日前、7日前に自動通知を送信する。期限切れの計画は「要更新」ステータスに自動変更する。
5. **配布管理**: 承認済み計画を指定した配布先（ユーザ/グループ）に配布する。既読確認機能を提供する。
6. **検索・フィルタ**: 全文検索、対象システム別、シナリオ別、ステータス別、期限別のフィルタリング。

#### 出力

| 出力項目 | 形式 | 説明 |
|---------|------|------|
| 計画文書一覧 | テーブル表示 | フィルタ・ソート・ページング対応 |
| 計画文書詳細 | 詳細画面 | 本文表示、メタデータ、変更履歴 |
| 計画文書PDF | PDF | 印刷用レイアウトでのPDF出力 |
| 変更履歴 | テーブル表示 | バージョン毎の変更内容と変更者 |
| 差分表示 | 差分ビュー | 2バージョン間の差分をハイライト表示 |
| 配布状況 | テーブル表示 | 配布先の既読/未読状況 |

#### 画面イメージ

1. **計画一覧画面**: カード形式またはテーブル形式の切替表示。ステータス別の色分け（ドラフト: グレー、レビュー中: 黄色、承認済み: 緑、期限切れ: 赤）。クイックフィルタ（対象システム、シナリオ、ステータス）をサイドパネルに配置。
2. **計画作成・編集画面**: 左側にMarkdownエディタ、右側にプレビューを配置するスプリットビュー。上部にメタデータ入力フォーム。下部に添付ファイルエリア。
3. **レビュー画面**: 計画本文にインラインコメントを付与可能。右サイドパネルにコメントスレッド一覧。上部に承認/差し戻しボタン。

#### 受け入れ基準

1. テンプレートから新規計画を作成できること
2. 計画の編集時に自動で版番号がインクリメントされること
3. レビュー・承認ワークフローが正しく動作すること（ステータス遷移）
4. 差し戻し時に理由が記録され、通知が送信されること
5. 有効期限の自動通知が正しいタイミングで送信されること
6. PDF出力が正しいレイアウトで生成されること
7. 全文検索が1秒以内に結果を返すこと
8. 版間の差分表示が正確であること
9. オフライン時に既存計画の閲覧が可能であること

### PLAN-002: RTO/RPO管理

| 項目 | 内容 |
|------|------|
| 要件ID | PLAN-002 |
| 要件名 | RTO/RPO管理 |
| 優先度 | 必須 |
| 概要 | 各ITシステムのRTO（目標復旧時間）およびRPO（目標復旧時点）を定義・管理し、達成状況を追跡する |

#### 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| システム名 | 選択 | 必須 | 対象システム（マスタから選択） |
| システムカテゴリ | 選択 | 必須 | インフラ/アプリケーション/SaaS等 |
| RTO | 時間（時:分） | 必須 | 目標復旧時間 |
| RPO | 時間（時:分） | 必須 | 目標復旧時点 |
| MTPD | 時間（時:分） | 必須 | 最大許容停止時間 |
| 復旧手順概要 | テキスト | 必須 | 復旧手順の概要説明 |
| 復旧担当チーム | 選択（複数可） | 必須 | 復旧を担当するチーム |
| 依存システム | 選択（複数可） | 任意 | 復旧に必要な前提システム |
| バックアップ方式 | 選択 | 必須 | フル/差分/増分/レプリケーション |
| バックアップ頻度 | テキスト | 必須 | バックアップの実施頻度 |
| 最終テスト日 | 日付 | 任意 | RTO/RPO達成の最終テスト日 |
| テスト結果 | 選択 | 任意 | 合格/不合格/未テスト |
| 根拠（BIA参照） | 選択 | 必須 | BIAの評価結果との紐づけ |

#### 処理

1. **RTO/RPO登録・更新**: 各システムのRTO/RPO値を登録・更新する。変更時は変更理由の記録を必須とする。
2. **依存関係管理**: システム間の依存関係を有向グラフとして管理する。復旧順序の整合性チェック（依存先のRTOが依存元のRTOより大きい場合に警告）を行う。
3. **達成状況追跡**: 訓練・実インシデントでのRTO/RPO達成実績を記録し、定義値との比較を行う。
4. **自動アラート**: RTO/RPOの定義値と実績値の乖離が閾値を超えた場合にアラートを発する。最終テスト日から一定期間（デフォルト90日）経過した場合に再テストを促す通知を送信する。
5. **BIA連動**: BIA結果に基づくRTO/RPO推奨値を自動計算し、現在の定義値との比較を表示する。

#### 出力

| 出力項目 | 形式 | 説明 |
|---------|------|------|
| RTO/RPO一覧 | テーブル表示 | 全システムのRTO/RPO定義一覧 |
| 依存関係図 | グラフ表示 | システム間の依存関係を可視化 |
| 達成状況マトリクス | ヒートマップ | RTO/RPO達成状況を色分けで表示 |
| 乖離レポート | テーブル + グラフ | 定義値と実績値の乖離を表示 |
| RTO/RPO定義書（PDF） | PDF | 全システムのRTO/RPO定義を出力 |

#### 画面イメージ

1. **RTO/RPO一覧画面**: テーブル形式で全システムを一覧表示。RTO/RPO列は色分け（短い: 赤、中程度: 黄、長い: 緑）。フィルタ（カテゴリ別、テスト状況別）、ソート対応。
2. **依存関係図画面**: インタラクティブなネットワークグラフ。ノード（システム）をクリックで詳細表示。依存方向を矢印で表示。不整合箇所をハイライト。
3. **RTO/RPO詳細画面**: 上部にシステム基本情報、中央にRTO/RPOの定義値と実績値の比較グラフ、下部にテスト履歴。

#### 受け入れ基準

1. 全対象システムのRTO/RPOを登録・管理できること
2. 依存関係の不整合を自動検知して警告すること
3. 訓練・実インシデントの実績データと定義値の比較が可能なこと
4. 最終テスト日に基づく再テスト通知が正しく動作すること
5. BIA結果との連動によるRTO/RPO推奨値が表示されること
6. オフライン時にRTO/RPO一覧の閲覧が可能であること

### PLAN-003: 代替手段管理

| 項目 | 内容 |
|------|------|
| 要件ID | PLAN-003 |
| 要件名 | 代替手段管理 |
| 優先度 | 必須 |
| 概要 | ITシステム停止時の代替運用手段（手動運用、代替システム等）を定義・管理する |

#### 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| 対象システム | 選択 | 必須 | 代替手段の対象システム |
| 代替手段名 | テキスト（最大200文字） | 必須 | 代替手段の名称 |
| 代替手段種別 | 選択 | 必須 | 手動運用/代替システム/外部サービス/縮退運転 |
| 発動条件 | テキスト | 必須 | 代替手段を発動する条件 |
| 手順書 | リッチテキスト | 必須 | 代替運用の詳細手順 |
| 制約事項 | テキスト | 任意 | 代替運用時の制約・注意事項 |
| 最大継続時間 | 時間 | 必須 | 代替手段で運用可能な最大時間 |
| 必要リソース | テキスト | 必須 | 必要な人員・機材・場所等 |
| 訓練頻度 | 選択 | 必須 | 代替手段の訓練頻度（月次/四半期/半年/年次） |
| 最終訓練日 | 日付 | 任意 | 最後に訓練を実施した日付 |
| 担当者 | ユーザ選択（複数可） | 必須 | 代替運用の実施担当者 |

#### 処理

1. **代替手段登録**: 各システムに対して1つ以上の代替手段を登録する。複数の代替手段を優先順位付きで登録可能。
2. **手順書管理**: 代替運用手順を詳細に記録。スクリーンショットや図表を含むリッチテキスト対応。チェックリスト形式での手順記述も対応。
3. **訓練スケジュール連動**: 代替手段の訓練頻度に基づいて訓練管理（TRN）機能と連動し、訓練スケジュールを自動提案する。
4. **ステータス管理**: 代替手段の状態を管理する（有効/レビュー中/期限切れ/無効）。最終訓練日から訓練頻度を超えた場合に「要訓練」アラートを発する。
5. **インシデント連動**: インシデント発生時に対象システムの代替手段を自動表示する。

#### 出力

| 出力項目 | 形式 | 説明 |
|---------|------|------|
| 代替手段一覧 | テーブル表示 | システム別の代替手段一覧 |
| 代替手段詳細 | 詳細画面 | 手順書、制約事項等の詳細 |
| 代替手段チェックリスト | PDF | 印刷可能なチェックリスト形式 |
| 訓練状況マトリクス | テーブル表示 | 代替手段ごとの訓練状況 |

#### 画面イメージ

1. **代替手段一覧画面**: システム別にグルーピングされたテーブル。ステータス別の色分け表示。訓練状況のバッジ表示（訓練済み: 緑、要訓練: 赤）。
2. **代替手段詳細画面**: 上部に基本情報、中央に手順書（チェックリスト形式対応）、右サイドに関連情報（訓練履歴、担当者）。
3. **インシデント時表示**: インシデント対応画面から呼び出される代替手段パネル。該当システムの代替手段をカード形式で表示。

#### 受け入れ基準

1. 各システムに対して複数の代替手段を登録・管理できること
2. 手順書にリッチテキスト（画像・表・チェックリスト含む）を記述できること
3. 訓練スケジュールとの連動が正しく動作すること
4. インシデント発生時に該当システムの代替手段が自動表示されること
5. オフライン時に代替手段の手順書を閲覧できること
6. チェックリスト形式でのPDF出力が可能であること

### PLAN-004: 連絡網管理

| 項目 | 内容 |
|------|------|
| 要件ID | PLAN-004 |
| 要件名 | 連絡網管理 |
| 優先度 | 必須 |
| 概要 | 緊急時の連絡先、連絡手順、エスカレーションルートを管理する |

#### 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| 連絡先名 | テキスト | 必須 | 連絡先の氏名または名称 |
| 所属 | テキスト | 必須 | 部署・組織名 |
| 役割 | 選択 | 必須 | BCP上の役割（指揮官/対応者/連絡担当/報告先等） |
| 電話番号（主） | 電話番号 | 必須 | 主連絡先電話番号 |
| 電話番号（副） | 電話番号 | 任意 | 副連絡先電話番号 |
| メールアドレス | メール | 必須 | メールアドレス |
| Teams/チャット | テキスト | 任意 | Microsoft Teams等のアカウント |
| 連絡可能時間帯 | 時間範囲 | 任意 | 通常の連絡可能時間帯 |
| 代理連絡先 | 連絡先選択 | 任意 | 本人に連絡がつかない場合の代理 |
| エスカレーション先 | 連絡先選択 | 任意 | エスカレーション先 |
| エスカレーション条件 | テキスト | 任意 | エスカレーションする条件 |
| 連絡網グループ | 選択（複数可） | 必須 | 所属する連絡網グループ |

#### 処理

1. **連絡先管理**: 緊急連絡先の登録・更新・削除。Active Directory / Entra IDとの同期による自動更新。手動での追加情報（私用携帯等）の登録。
2. **連絡網ツリー管理**: 階層型の連絡網ツリーを視覚的に編集可能。シナリオ別（地震、サイバー攻撃等）の連絡網パターンを設定。
3. **エスカレーション管理**: エスカレーションルートを定義。時間経過に基づく自動エスカレーション（例: 15分応答なしで上位にエスカレーション）。
4. **連絡実行支援**: ワンクリックでの電話発信（tel:リンク）、メール送信、Teams通知。一斉連絡機能（テンプレートベース）。連絡結果（応答/不応答/代理応答）の記録。
5. **定期更新促進**: 四半期ごとに連絡先の確認依頼を自動送信。未確認の連絡先を「要確認」ステータスで表示。
6. **Entra ID連携**: Entra IDのユーザ情報と同期し、異動・退職等の変更を自動反映。

#### 出力

| 出力項目 | 形式 | 説明 |
|---------|------|------|
| 連絡網ツリー | ツリー図 | 階層型連絡網の視覚表示 |
| 連絡先一覧 | テーブル表示 | グループ別・役割別の連絡先一覧 |
| 連絡カード | カード表示 | 個人の連絡先詳細 |
| 連絡網PDF | PDF | 印刷用連絡網 |
| 連絡履歴 | テーブル表示 | 連絡実行の結果一覧 |

#### 画面イメージ

1. **連絡網ツリー画面**: ドラッグ&ドロップで編集可能な組織図形式。ノードに連絡先情報のサマリーを表示。シナリオ切替タブ。
2. **連絡先一覧画面**: 検索・フィルタ対応のテーブル。ワンクリック連絡ボタン（電話/メール/Teams）。ステータス表示（確認済み/要確認）。
3. **一斉連絡画面**: テンプレート選択、送信先グループ選択、メッセージカスタマイズ、送信確認のウィザード形式。

#### 受け入れ基準

1. 連絡先の登録・更新・削除が正常に行えること
2. Entra IDとの同期が正しく動作すること
3. 連絡網ツリーをドラッグ&ドロップで編集できること
4. ワンクリックでの電話発信・メール送信・Teams通知が動作すること
5. 一斉連絡機能が正しく動作し、送信結果が記録されること
6. オフライン時に連絡先情報が閲覧できること
7. 連絡先の定期確認通知が正しく送信されること
8. 個人情報の適切な保護（アクセス制御）が実装されていること

### PLAN-005: 優先復旧順序管理

| 項目 | 内容 |
|------|------|
| 要件ID | PLAN-005 |
| 要件名 | 優先復旧順序管理 |
| 優先度 | 必須 |
| 概要 | ITシステムの復旧優先順位を定義・管理し、復旧計画の整合性を確保する |

#### 入力

| 入力項目 | データ型 | 必須 | 説明 |
|---------|---------|------|------|
| システム名 | 選択 | 必須 | 対象システム |
| 復旧優先度 | 選択 | 必須 | Tier1（最優先）/ Tier2（高優先）/ Tier3（中優先）/ Tier4（低優先） |
| 復旧フェーズ | 選択 | 必須 | 即時（0-4h）/ 短期（4-24h）/ 中期（24-72h）/ 長期（72h-） |
| 依存先システム | 選択（複数可） | 任意 | 復旧前に稼働している必要があるシステム |
| 復旧チーム | 選択 | 必須 | 復旧を担当するチーム |
| 前提条件 | テキスト | 任意 | 復旧開始の前提条件 |
| 備考 | テキスト | 任意 | 特記事項 |

2. **依存関係検証**: システム間の依存関係を考慮し、復旧順序の整合性を自動検証。循環依存や逆順依存（依存先が依存元より低い優先度）を検知して警告。

#### 受け入れ基準

#### 受け入れ基準

1. インシデント宣言から5秒以内にコマンドセンターが起動すること

5. タイムラインにイベントがタイムスタンプ付きで記録されること
6. モバイル端末からコマンドセンターにアクセスできること
7. オフライン時に基本的な参照機能（手順、連絡先）が利用できること

2. **詳細ドリルダウン**: カテゴリをクリックで詳細スコアの内訳を表示。各項目の充足/未充足を色分け表示。

#### 受け入れ基準

1. レディネススコアが自動計算されること
2. スコアの算出根拠が確認できること（ドリルダウン）
3. 改善提案が自動生成されること
4. トレンドグラフが正しく表示されること
