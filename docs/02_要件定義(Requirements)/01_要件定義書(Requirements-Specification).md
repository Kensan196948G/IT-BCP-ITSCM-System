## 1. はじめに

### 1.1 本書の目的

本書は、みらい建設工業株式会社向け「IT事業継続管理システム（IT-BCP-ITSCM-System）」の要件定義書である。本システムの開発に必要な要件を体系的に定義し、ステークホルダー間の合意形成と開発チームへの正確な要件伝達を目的とする。

### 1.2 本書の読者

- プロジェクトオーナー / スポンサー

- 品質保証（QA）チーム
- インフラ / 運用チーム
- 情報セキュリティ担当者
- 業務部門の利用者代表

本書は以下のドキュメント群の上位に位置し、各詳細ドキュメントの根拠となる。

## 2. プロジェクト概要

### 2.1 プロジェクト名

IT事業継続管理システム（IT-BCP-ITSCM-System）

### 2.2 プロジェクトの概要

本プロジェクトは、災害・サイバー攻撃等の有事発生時におけるIT復旧計画の管理、BCP訓練の実施・記録、およびRTO（目標復旧時間）ダッシュボードを統合した事業継続管理プラットフォームを構築するものである。

建設業特有の要件（現場作業員のモバイル利用、通信環境が不安定な建設現場でのオフライン動作、多拠点にまたがるプロジェクト管理等）を考慮し、PWA（Progressive Web App）による完全オフライン動作を実現する。

### 2.3 対象組織

みらい建設工業株式会社（以下「みらい建設」）

### 2.4 技術スタック

| レイヤー | 技術 |
|---------|------|
| フロントエンド | Next.js 14 / TypeScript / PWA対応 |
| バックエンド | Python FastAPI |
| データベース | PostgreSQL |
| キャッシュ / メッセージブローカー | Redis |
| 非同期タスク処理 | Celery |
| インフラ | Azure Container Apps |
| リージョン構成 | マルチリージョン（東日本 + 西日本） |

## 3. 目的と背景

### 3.1 背景

近年、建設業界においてもITシステムへの依存度が急速に高まっている。みらい建設では以下の環境変化に直面している。

1. **自然災害リスクの増大**: 南海トラフ地震、首都直下型地震等の大規模地震リスク。建設現場は全国に分散しており、広域災害時の事業継続が喫緊の課題である。

2. **サイバー攻撃の高度化**: 建設業界を標的としたランサムウェア攻撃が増加。設計データ、施工管理データ等の重要情報資産の保護と迅速な復旧体制が求められている。

3. **規制・コンプライアンス要件の強化**: ISO20000（ITサービスマネジメント）におけるITSCM（ITサービス継続性管理）の実装が求められており、監査対応のための文書化・証跡管理が必要である。

4. **既存のBCP管理の課題**: 現在のBCP関連文書はExcel・Word等で個別に管理されており、リアルタイムでの状況把握、訓練結果の体系的な管理、復旧計画の整合性維持が困難である。

5. **現場の機動性確保**: 建設現場では通信環境が不安定な場合が多く、災害時はさらに通信障害が発生する可能性が高い。オフラインでも機能する緊急対応ツールが不可欠である。

### 3.2 目的

本システムの構築により、以下の目的を達成する。

1. **IT復旧計画の一元管理**: IT復旧計画、RTO/RPO定義、代替手段、連絡網等を統合的に管理し、常に最新の状態を維持する。

2. **BCP訓練の効率化と品質向上**: テーブルトップ演習や実機テストの計画・実施・評価をシステム化し、訓練の質と頻度を向上させる。

3. **インシデント発生時の迅速な対応**: 緊急時の指揮統制、タスク管理、状況報告を支援し、RTOの達成を確実にする。

4. **コンプライアンスの確保**: ISO20000 ITSCM、ISO27001、NIST CSFの要件を満たし、監査対応の効率化を図る。

5. **レジリエンスの可視化**: BCPレディネスダッシュボード、RTO/RPOコンプライアンスレポート等によりIT事業継続管理の現状を可視化し、継続的改善を促進する。

6. **オフライン対応の実現**: PWA技術により、通信障害時でもシステムの基本機能を利用可能とする。

### 3.3 期待される効果

| 効果項目 | 定量目標 |
|---------|---------|
| RTO達成率 | 95%以上（訓練時） |
| BCP訓練実施頻度 | 年4回以上（四半期ごと） |
| 復旧計画の更新頻度 | 四半期ごと + 変更発生時 |
| インシデント対応初動時間 | 15分以内 |
| ISO20000 ITSCM監査指摘事項 | 0件（重大指摘） |
| オフライン時の基本機能利用率 | 100%（緊急連絡・手順参照） |

## 4. スコープ

### 4.1 スコープ内

1. **BCP/ITSCM計画管理機能**: IT復旧計画の作成・承認・版管理、RTO/RPO定義管理、代替手段管理、連絡網管理、優先復旧順序管理、ベンダー連絡先管理

2. **BCP訓練管理機能**: 訓練シナリオ作成・管理、テーブルトップ演習の実施支援、訓練結果記録・評価、タイムトラッキング、訓練レポート自動生成、実機テスト計画管理

3. **インシデント対応機能**: 緊急対応指揮支援（コマンドセンター）、RTOリアルタイムダッシュボード、タスク割当・進捗管理、状況報告の自動化、外部連絡管理（規制当局・取引先）、モバイル対応

4. **ダッシュボード・レポート機能**: BCPレディネスダッシュボード、RTO/RPOコンプライアンスレポート、訓練履歴・改善トレンド分析、ISO20000 ITSCM準拠レポート

5. **ビジネスインパクト分析（BIA）機能**: 業務プロセス影響評価、財務影響試算、規制違反リスク評価、評価結果の可視化

6. **システム基盤**: PWA対応（オフライン動作）、マルチリージョン冗長構成、本システム自体のBCP（RTO 15分以内）

### 4.2 スコープ外

1. 一般的なITSM機能（インシデント管理、問題管理、変更管理等）は既存ITSMシステムとの連携で対応
2. ネットワーク監視・サーバー監視機能（既存SIEM/監視基盤との連携で対応）
3. バックアップ・リストアの実行機能（既存バックアップ基盤との連携で対応）
4. 人事・労務管理機能
5. 会計・経理機能
6. 建設プロジェクト管理機能（工程管理、原価管理等）

### 4.3 外部システム連携

| 連携先システム | 連携内容 | 連携方式 |
|--------------|---------|---------|
| Active Directory / Entra ID | 認証・認可 | SAML 2.0 / OIDC |
| ITSM-System | インシデント情報連携 | REST API |
| SIEM Platform | セキュリティイベント連携 | REST API / Webhook |
| Exchange Online | メール通知・連絡網 | Microsoft Graph API |
| Microsoft Teams | チャット通知・緊急連絡 | Microsoft Graph API / Webhook |
| Azure Monitor | システム監視データ連携 | Azure SDK |

| インシデント対応者 | インシデント指揮官の指示のもと、実際の復旧作業を実行する技術者 | 10-20名 |

| 閲覧ユーザ | BCPに関する情報を参照する一般ユーザ。緊急時連絡先、手順書の閲覧 | 50-100名 |

### 5.2 利用シーン

#### 平時

- BCP計画の作成・更新・レビュー・承認
- BCP訓練の企画・実施・評価
- BIAの実施と更新
- ダッシュボード・レポートの確認
- 監査対応資料の出力

#### 有事（インシデント発生時）

- 緊急対応コマンドセンターの起動
- RTOダッシュボードでの復旧状況モニタリング
- タスク割当と進捗管理
- 状況報告の作成・配信
- 外部連絡（規制当局・取引先・ベンダー）
- モバイル端末からの操作（建設現場等）

#### オフライン時

- 緊急連絡先の参照
- 復旧手順書の参照
- チェックリストの確認・記録
- オンライン復帰時のデータ同期

## 6. 用語定義

| 用語 | 定義 |
|------|------|
| BCP (Business Continuity Plan) | 事業継続計画。災害等の緊急事態に遭遇した場合に、事業資産の損害を最小限にとどめつつ、中核となる事業の継続あるいは早期復旧を可能とするために、平常時に行うべき活動や緊急時における事業継続のための方法、手段等を取り決めておく計画 |
| ITSCM (IT Service Continuity Management) | ITサービス継続性管理。ITサービスの中断からの復旧を管理するITSMプロセス。ISO20000で規定 |
| RTO (Recovery Time Objective) | 目標復旧時間。災害等により業務が中断した際に、どのくらいの時間で復旧させるかの目標値 |
| RPO (Recovery Point Objective) | 目標復旧時点。災害等によりデータが失われた際に、どの時点のデータまで復旧させるかの目標値 |
| BIA (Business Impact Analysis) | ビジネスインパクト分析。業務中断による影響を分析し、重要業務の特定とRTO/RPOの設定根拠を明確にする分析手法 |
| MTPD (Maximum Tolerable Period of Disruption) | 最大許容停止時間。事業の存続が脅かされるまでの最大停止時間 |
| PWA (Progressive Web App) | プログレッシブウェブアプリ。Webアプリをネイティブアプリのように利用できる技術。オフライン動作、プッシュ通知等を実現 |
| テーブルトップ演習 | 机上演習。実際のシステムを使わず、シナリオに基づいて関係者が議論形式で復旧手順を確認する訓練方法 |
| フェイルオーバー | 稼働中のシステムに障害が発生した際に、待機系のシステムに自動的に切り替えること |
| マルチリージョン | 複数の地理的に離れたデータセンターリージョンにシステムを配置し、耐障害性を高める構成 |
| コマンドセンター | 有事発生時に緊急対応を指揮・統制するための仮想的な司令室。本システムではWeb上で実現 |
| レジリエンス | 障害や災害からの回復力。システムや組織が困難な状況から迅速に復旧し、正常な状態に戻る能力 |
| NIST CSF | 米国国立標準技術研究所（NIST）が策定したサイバーセキュリティフレームワーク。RECOVER（復旧）機能を含む |
| ISO20000 | ITサービスマネジメントの国際規格。ITSCMはその中のプロセスの一つ |
| ISO27001 | 情報セキュリティマネジメントシステム（ISMS）の国際規格。A.5.29（事業継続計画におけるICT）、A.5.30（事業継続のためのICTの備え）を含む |

## 7. 対象シナリオ

本システムが対応すべき事業継続シナリオを以下に定義する。

### 7.1 大規模地震

| 項目 | 内容 |
|------|------|
| シナリオID | SCN-001 |
| シナリオ名 | 大規模地震（震度6強以上） |
| 想定事象 | 南海トラフ地震、首都直下型地震等による広域被害 |
| 影響範囲 | 全社拠点、建設現場、データセンター |
| 主な影響 | 電力停止、通信障害、建物損壊、交通遮断、人員被害 |
| ITへの影響 | DCの物理的損壊、電力・冷却喪失、ネットワーク断、クラウドアクセス不能 |
| 想定復旧期間 | 数時間〜数週間 |
| 対応優先度 | 最高 |

### 7.2 サイバー攻撃（ランサムウェア）

| 項目 | 内容 |
|------|------|
| シナリオID | SCN-002 |
| シナリオ名 | ランサムウェア攻撃 |
| 想定事象 | ランサムウェアによる全社的なシステム暗号化・データ暗号化 |
| 影響範囲 | 社内IT環境全体 |
| 主な影響 | システム利用不能、データアクセス不能、業務停止 |
| ITへの影響 | サーバー・端末の暗号化、Active Directory侵害、バックアップ汚染 |
| 想定復旧期間 | 数日〜数週間 |
| 対応優先度 | 最高 |

### 7.3 データセンター障害

| 項目 | 内容 |
|------|------|
| シナリオID | SCN-003 |
| シナリオ名 | データセンター障害 |
| 想定事象 | 電力障害、空調故障、火災、物理的セキュリティ侵害 |
| 影響範囲 | 該当DCに設置された全システム |
| 主な影響 | DCホスティングサービスの全面停止 |
| ITへの影響 | オンプレミスシステムの停止、ネットワーク接続の喪失 |
| 想定復旧期間 | 数時間〜数日 |
| 対応優先度 | 高 |

### 7.4 クラウド広域障害

| 項目 | 内容 |
|------|------|
| シナリオID | SCN-004 |
| シナリオ名 | クラウド広域障害 |
| 想定事象 | Azureの広域障害、Microsoft 365の大規模障害 |
| 影響範囲 | クラウドサービスに依存する全業務 |
| 主な影響 | クラウドサービス利用不能、メール・コラボレーション停止 |
| ITへの影響 | SaaS/PaaS/IaaSの全面停止、認証基盤への影響 |
| 想定復旧期間 | 数時間〜1日（クラウド事業者の復旧に依存） |
| 対応優先度 | 高 |

### 7.5 パンデミック

| 項目 | 内容 |
|------|------|
| シナリオID | SCN-005 |
| シナリオ名 | パンデミック |
| 想定事象 | 新型感染症の大規模流行 |
| 影響範囲 | 全社の人的リソース |
| 主な影響 | 人員不足（出社率30-50%）、リモートワーク環境への負荷集中 |
| ITへの影響 | VPN・リモートアクセス環境への負荷増大、IT要員の不足 |
| 想定復旧期間 | 数週間〜数ヶ月 |
| 対応優先度 | 中 |

### 7.6 サプライヤー障害

| 項目 | 内容 |
|------|------|
| シナリオID | SCN-006 |
| シナリオ名 | サプライヤー障害 |
| 想定事象 | ITベンダー・通信事業者の重大障害、契約終了、倒産 |
| 影響範囲 | 該当ベンダーに依存するシステム・サービス |
| 主な影響 | 保守サポート喪失、サービス停止、データ移行の必要性 |
| ITへの影響 | ベンダー管理システムの停止、SLA違反 |
| 想定復旧期間 | 数日〜数ヶ月 |
| 対応優先度 | 中 |

### 7.7 局所的システム障害

| 項目 | 内容 |
|------|------|
| シナリオID | SCN-007 |
| シナリオ名 | 局所的システム障害 |
| 想定事象 | 個別システムのハードウェア障害、ソフトウェア障害、設定ミス |
| 影響範囲 | 該当システムおよび依存システム |
| 主な影響 | 特定業務の停止 |
| ITへの影響 | 個別システムの停止、データ不整合 |
| 想定復旧期間 | 数分〜数時間 |
| 対応優先度 | 中〜高（対象システムの重要度による） |

## 12. 前提条件・制約条件

### 12.1 前提条件

1. みらい建設は Microsoft 365 E5 ライセンスを保有しており、Microsoft Entra ID（旧Azure AD）を認証基盤として利用可能である
2. Azure サブスクリプションが利用可能であり、東日本・西日本リージョンでのリソース作成が許可されている
3. 社内ネットワークからAzureへの接続（VPN / ExpressRoute）が確立されている
4. 既存のITSMシステム、SIEMプラットフォームとのAPI連携が技術的に可能である
5. 社内のBCP関連文書（現行のExcel/Word版）が要件定義時に提供される
6. プロジェクトに必要な開発・運用リソースが確保される

### 12.2 制約条件

1. **予算制約**: プロジェクト予算は承認済み予算の範囲内とする
2. **スケジュール制約**: 2026年度内（2027年3月末）までに本番稼働開始
3. **技術制約**: みらい建設のITガバナンスポリシーに準拠した技術選定を行う
4. **データ制約**: 個人情報（連絡先等）の取扱いは個人情報保護法および社内規程に準拠する
5. **運用制約**: 本番運用は既存の情報システム部門体制で対応可能な運用負荷とする
6. **リージョン制約**: データは日本国内のAzureリージョンにのみ保存する（データレジデンシー）
