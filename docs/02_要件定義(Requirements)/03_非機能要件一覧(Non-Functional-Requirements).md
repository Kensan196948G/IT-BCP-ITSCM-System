## 1. 可用性要件

### NFR-AVL-001: システム稼働率

| 項目 | 内容 |
|------|------|
| 要件ID | NFR-AVL-001 |
| 要件名 | システム稼働率 |
| 優先度 | 必須 |
| 目標値 | 99.9%（計画停止を除く） |

#### 詳細

| 測定項目 | 目標値 | 計測方法 |
|---------|--------|---------|
| 年間稼働率 | 99.9% | Azure Monitor による稼働監視 |
| 年間最大停止時間 | 8時間45分以内（計画停止除く） | 障害レポートから集計 |
| 計画停止 | 月1回、最大2時間以内 | 計画停止スケジュール管理 |
| 計画停止通知 | 5営業日前に通知 | メール/Teams通知 |
| 緊急メンテナンス | 年間4回以内 | 緊急メンテナンスログ |

#### 実装方針

- Azure Container Appsの複数レプリカ構成（最小2レプリカ）
- Azure Front Doorによるグローバルロードバランシング
- 東日本（Primary）・西日本（Secondary）のマルチリージョン構成
- ヘルスチェックによる異常検知と自動復旧（コンテナ再起動）
- データベースのフェイルオーバー構成（Azure Database for PostgreSQL Flexible Server - ゾーン冗長HA）

### NFR-AVL-002: 本システムのRTO/RPO

| 項目 | 内容 |
|------|------|
| 要件ID | NFR-AVL-002 |
| 要件名 | 本システムのRTO/RPO |
| 優先度 | 必須 |
| RTO | 15分以内 |
| RPO | 5分以内 |

#### 詳細

| 障害シナリオ | RTO目標 | RPO目標 | 対策 |
|------------|---------|---------|------|
| 単一コンテナ障害 | 1分以内 | 0分（データ非依存） | 自動再起動、複数レプリカ |
| 単一リージョン障害 | 15分以内 | 5分以内 | リージョン間フェイルオーバー |
| データベース障害 | 10分以内 | 5分以内 | ゾーン冗長HA + リードレプリカ |
| Redis障害 | 5分以内 | 0分（再構築可能） | Redis Cluster + 自動フェイルオーバー |
| 全リージョン障害 | 60分以内 | 15分以内 | バックアップからの復元 |

#### 実装方針

- PostgreSQL: Azure Database for PostgreSQL Flexible Serverのゾーン冗長HA構成。リージョン間リードレプリカ。WAL（Write-Ahead Log）による5分RPOの確保。
- Redis: Azure Cache for Redisのゾーン冗長構成。セッション/キャッシュデータのため、RPOは0分（再構築可能）。
- コンテナ: Liveness/Readiness Probeによる異常検知。自動スケーリングと自動復旧。
- Front Door: 自動フェイルオーバーによるリージョン切替（DNS TTL: 30秒）。

### NFR-AVL-003: マルチリージョン冗長構成

| 項目 | 内容 |
|------|------|
| 要件ID | NFR-AVL-003 |
| 要件名 | マルチリージョン冗長構成 |
| 優先度 | 必須 |

#### 詳細

| 構成要素 | 東日本（Primary） | 西日本（Secondary） | 同期方式 |
|---------|------------------|-------------------|---------|
| Frontend (Next.js) | Active | Active | 同一コンテナイメージ |
| Backend (FastAPI) | Active | Standby (Warm) | 同一コンテナイメージ |
| PostgreSQL | Primary (Read/Write) | Replica (Read Only) | 非同期レプリケーション |
| Redis | Primary | Replica | 非同期レプリケーション |
| Celery Workers | Active | Standby | 同一コンテナイメージ |
| Blob Storage | Primary | GRS (Geo-Redundant) | Azureネイティブ |

#### フェイルオーバー手順

1. Azure Front Doorがプライマリリージョンの異常を検知（ヘルスプローブ失敗3回連続）
2. トラフィックをセカンダリリージョンに自動ルーティング
3. セカンダリのPostgreSQLレプリカをPrimaryに昇格（自動/手動切替可能）
4. セカンダリのBackend/Celeryを Active に切替
5. フェイルバック時はリバースレプリケーションで復旧

| 要件名 | 同時接続ユーザ数 |
| 優先度 | 必須 |

#### 詳細

| 条件 | 同時接続数 | 性能要件 |
|------|----------|---------|
| 通常時 | 50ユーザ | 全レスポンスタイム要件を満たす |
| ピーク時（訓練中） | 100ユーザ | レスポンスタイム目標の120%以内 |

- WebSocket: Redis PubSubによるスケールアウト対応
- CDN: 静的アセットのCDN配信（Azure CDN）
