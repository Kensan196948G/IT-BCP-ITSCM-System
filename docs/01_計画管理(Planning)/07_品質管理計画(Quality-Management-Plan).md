## 1. 目的

本計画は、IT-BCP-ITSCM-Systemプロジェクトにおける品質目標を定義し、その達成のための品質管理プロセス・品質指標（KPI）・レビュープロセス・テスト戦略を規定する。

---

## 2. 品質方針

### 2.1 基本方針

1. **品質はプロセスで作り込む**: テストだけでなく、設計・コードレビュー・CI/CDの各段階で品質を確保する
2. **自動化優先**: テスト・品質チェック・デプロイは可能な限り自動化し、人的ミスを排除する
3. **早期発見・早期修正**: 品質問題は開発サイクルの早い段階で検知し、修正コストを最小化する
4. **継続的改善**: レトロスペクティブ・メトリクス分析に基づく品質プロセスの継続的改善

## 4. 品質指標（KPI）

### 4.1 KPIダッシュボード

| カテゴリ | KPI | 計算式 | 目標 | 頻度 | 責任者 |
|---------|-----|--------|------|------|-------|
| コード品質 | テストカバレッジ（行） | テスト対象行 / 全行 | >=80% | 毎CI | QAリード |
| コード品質 | 静的解析違反数 | ESLint/Ruff違反数 | 0 | 毎CI | 各リード |
| コード品質 | 型安全性 | TypeScript strict / mypy通過率 | 100% | 毎CI | 各リード |
| コード品質 | コード重複率 | 重複行 / 全行 | <=3% | 週次 | アーキテクト |
| バグ管理 | オープンバグ数 | 未解決バグ合計 | Critical: 0, High: <=3 | 日次 | QAリード |
| バグ管理 | バグ密度 | バグ数 / KLOC | <=5 | スプリント | QAリード |
| バグ管理 | バグ修正率 | 修正済み / 報告済み | >=90% | スプリント | QAリード |
| テスト | テスト合格率 | パス数 / 全テスト数 | >=98% | 毎CI | QAリード |
| テスト | E2Eテストカバレッジ | テスト対象シナリオ / 全シナリオ | >=90% | スプリント | QAリード |
| 性能 | API P95レイテンシ | 95パーセンタイル | <=500ms | 週次 | SREリード |
| 性能 | ページLCP（P95） | 95パーセンタイル | <=2s | 週次 | FEリード |
| セキュリティ | 脆弱性スキャン結果 | Critical + High件数 | 0 | 毎CI | セキュリティEng |
| デプロイ | デプロイ頻度 | デプロイ回数 / 週 | >=2 | 週次 | SREリード |
| デプロイ | デプロイ失敗率 | 失敗数 / 全デプロイ数 | <=5% | 週次 | SREリード |

### 4.2 品質ゲート

CI/CDパイプラインに以下の品質ゲートを設定し、基準を満たさないコードはマージ・デプロイを阻止する。

| ゲート | 段階 | 基準 | 阻止対象 |
|--------|------|------|---------|
| Lint Gate | PR作成時 | ESLint/Ruff/Prettier違反0件 | PRマージ |
| Type Check Gate | PR作成時 | TypeScript strict / mypy エラー0件 | PRマージ |
| Unit Test Gate | PR作成時 | 全テストパス + カバレッジ80%以上 | PRマージ |
| Security Gate | PR作成時 | Dependabot/Snyk Critical/High 0件 | PRマージ |
| Build Gate | PR作成時 | ビルド成功 | PRマージ |
| Integration Test Gate | staging deploy後 | 結合テスト全パス | 本番デプロイ |
| E2E Test Gate | staging deploy後 | E2Eテスト全パス | 本番デプロイ |
| Performance Gate | staging deploy後 | P95レイテンシ基準内 | 本番デプロイ |

## 5. レビュープロセス

### 5.1 コードレビュー

#### 5.1.1 レビュー体制

| 対象 | レビュアー | 最低レビュー数 | 自動割当 |
|------|----------|-------------|---------|
| 通常のPR | 同チームメンバー | 1名以上 | CODEOWNERS |
| アーキテクチャ影響PR | アーキテクト + チームリード | 2名以上 | CODEOWNERS |
| セキュリティ関連PR | セキュリティエンジニア | 1名以上 | CODEOWNERS |
| インフラ変更PR | SREリード | 1名以上 | CODEOWNERS |

#### 5.1.2 レビュー観点チェックリスト

**機能面**
- [ ] 要件を正しく実装しているか
- [ ] エッジケース・異常系が考慮されているか
- [ ] APIのエラーハンドリングは適切か

**コード品質**
- [ ] 命名規約に従っているか
- [ ] 関数・クラスの責務が適切か（単一責任原則）
- [ ] コードの重複はないか（DRY原則）
- [ ] 適切にコメントされているか（必要な箇所のみ）

**セキュリティ**
- [ ] 入力値のバリデーションは実装されているか
- [ ] SQLインジェクション・XSS等の脆弱性はないか
- [ ] 機密情報がハードコードされていないか
- [ ] 認証・認可チェックは適切か

**テスト**
- [ ] 十分なテストが書かれているか
- [ ] テストは意味のあるアサーションを含んでいるか
- [ ] テストカバレッジが基準を満たしているか

**パフォーマンス**
- [ ] N+1クエリ等の性能問題はないか
- [ ] 不要な再レンダリングはないか（React）
- [ ] キャッシュの活用は適切か

#### 5.1.3 レビューSLA

| 優先度 | レビュー開始 | レビュー完了 |
|-------|------------|------------|
| 緊急（本番障害修正） | 30分以内 | 2時間以内 |
| 高（スプリント目標に影響） | 4時間以内 | 24時間以内 |
| 通常 | 24時間以内 | 48時間以内 |

| レビュー種別 | タイミング | 参加者 | レビュー対象 |
|------------|----------|--------|------------|

## 6. テスト戦略

### 6.1 テストレベル

```
            /\
           /  \
          / E2E \        ← 少数の重要シナリオ
         / Tests \
        /----------\
       / Integration \    ← API・DB連携テスト
      /    Tests      \
     /------------------\
    /    Unit Tests      \  ← 大量の高速テスト
   /______________________\

         テストピラミッド
```

### 6.2 テストレベル詳細

#### 6.2.1 単体テスト（Unit Test）

| 項目 | 内容 |
|------|------|
| 目的 | 個々の関数・コンポーネントの正確性検証 |
| 対象 | フロントエンド: Reactコンポーネント、ユーティリティ関数、Hooks / バックエンド: API ハンドラー、ビジネスロジック、ユーティリティ |
| ツール | FE: Vitest + React Testing Library / BE: pytest |
| カバレッジ目標 | 80%以上（行カバレッジ） |
| 実行タイミング | 毎コミット（CI） |
| 責任者 | 各開発者 |

#### 6.2.2 結合テスト（Integration Test）

| 項目 | 内容 |
|------|------|
| 目的 | コンポーネント間の連携・API-DB連携の検証 |
| 対象 | API エンドポイント（DB含む）、認証フロー、ワークフロー |
| ツール | BE: pytest + httpx + testcontainers |
| 実行タイミング | 毎PR（CI）、Staging デプロイ後 |
| 責任者 | QAチーム + 開発者 |

#### 6.2.3 E2Eテスト（End-to-End Test）

| 項目 | 内容 |
|------|------|
| 目的 | ユーザーシナリオに基づく一気通貫の検証 |
| 対象 | 主要ユーザーフロー（計画作成→承認、訓練実施→評価 等） |
| ツール | Playwright |
| シナリオ数 | 各機能の主要フロー + 異常系（合計50-80シナリオ想定） |
| 実行タイミング | Staging デプロイ後（日次） |
| 責任者 | QAチーム |

#### 6.2.4 性能テスト（Performance Test）

| 項目 | 内容 |
|------|------|
| 目的 | 負荷条件下での性能・安定性検証 |
| テスト種別 | 負荷テスト、ストレステスト、耐久テスト |
| ツール | k6 / Locust |
| 条件 | 同時接続100ユーザー、ピーク時200ユーザー |
| 基準 | API P95 <= 500ms、ページP95 <= 2s、エラー率 <= 0.1% |
| 実行タイミング | Phase 3完了時、Phase 4（本番デプロイ前） |
| 責任者 | QAチーム + SREチーム |

#### 6.2.5 セキュリティテスト

| 項目 | 内容 |
|------|------|
| 目的 | セキュリティ脆弱性の検出・排除 |
| テスト種別 | SAST（静的）、DAST（動的）、ペネトレーションテスト |
| ツール | SAST: Snyk/Bandit、DAST: OWASP ZAP、依存関係: Dependabot |
| 基準 | Critical/High脆弱性: 0件 |
| 実行タイミング | SAST: 毎CI、DAST: Phase 2以降週次、ペンテスト: Phase 4 |
| 責任者 | セキュリティエンジニア |

#### 6.2.6 アクセシビリティテスト

| 項目 | 内容 |
|------|------|
| 目的 | WCAG 2.1 AA準拠の検証 |
| ツール | axe-core（CI統合）、Lighthouse |
| 基準 | axe-core違反0件、Lighthouseアクセシビリティスコア90以上 |
| 実行タイミング | 毎PR（axe-core）、Phase 3（Lighthouse） |
| 責任者 | FEチーム |

### 6.3 テスト環境

| 環境 | 用途 | 構成 | データ |
|------|------|------|--------|
| ローカル | 開発・単体テスト | Docker Compose | モックデータ |
| CI | 自動テスト（単体・結合） | GitHub Actions | テストフィクスチャ |
| Staging | 結合テスト・E2E・性能テスト | Azure（東日本） | テストデータ（匿名化） |
| 本番 | 本番運用 | Azure（東日本+西日本） | 本番データ |

### 6.4 バグ管理

#### 6.4.1 バグ重要度分類

| 重要度 | 定義 | 修正期限 | 例 |
|--------|------|---------|---|
| Critical | システム全体が利用不可 | 即時（24時間以内） | 全ユーザーのログイン不可 |
| High | 主要機能が利用不可（回避策なし） | 3営業日以内 | BCP計画の保存ができない |
| Medium | 機能の一部が利用不可（回避策あり） | 現スプリント内 | フィルタ条件が正しく動作しない |
| Low | 軽微なUI問題・改善要望 | 次スプリント以降 | レイアウトの微調整 |

#### 6.4.2 バグ報告テンプレート

```
## バグ報告
- タイトル: [簡潔な問題の説明]
- 重要度: [Critical / High / Medium / Low]
- 環境: [ローカル / Staging / 本番]
- ブラウザ/OS: [例: Chrome 120 / Windows 11]

### 再現手順
1. ...
2. ...
3. ...

### 期待される動作
...

### 実際の動作
...

### スクリーンショット/ログ
...

### 関連情報
- 関連PR/Issue: #xxx
```
