## 1. 概要

### 1.1 目的

本文書は、IT-BCP-ITSCMシステムの可用性、パフォーマンス、セキュリティを確保するための監視設計を定義する。問題の早期検知、迅速な対応、データに基づく改善を実現する。

### 1.2 監視方針

| 方針 | 内容 |
|------|------|
| 多層監視 | インフラ、アプリケーション、ビジネスの各層を監視 |
| プロアクティブ監視 | 異常の予兆を検知し、障害を未然に防止 |
| 自動化 | アラートの自動発報、一部の自動復旧 |
| 可視化 | リアルタイムダッシュボードによる状況把握 |
| SLA/SLO駆動 | SLA/SLO目標に基づくアラート設計 |

## 4. 監視ツール

### 4.1 Azure Monitor

```
用途: Azureリソースのメトリクス・ログ収集、アラート管理

設定:
  - 診断設定: 全リソースでLog Analyticsへの送信有効化
  - メトリクスアラート: リソースメトリクスベースのアラート
  - ログアラート: KQLベースのカスタムアラート
  - アクションルール: アラート抑制・ルーティング
```

### 4.2 Application Insights

```
用途: APM（Application Performance Monitoring）

設定:
  バックエンド（Python FastAPI）:
    - OpenCensus Azure Monitor Exporter
    - 自動依存関係追跡（DB、Redis、HTTP）
    - カスタムメトリクス送信
    - サンプリングレート: 本番100%

  フロントエンド（Next.js）:
    - @microsoft/applicationinsights-web
    - ページビュー自動追跡
    - Ajaxリクエスト追跡
    - JavaScriptエラー追跡
    - ユーザーセッション追跡

  機能:
    - Application Map（依存関係可視化）
    - Live Metrics（リアルタイムモニタリング）
    - Transaction Search（トランザクション検索）
    - Failures（エラー分析）
    - Performance（パフォーマンス分析）
    - Users/Sessions/Events（ユーザー分析）
    - Availability Tests（可用性テスト）
```

## 5. アラート設計

### 5.1 アラート重要度定義

| 重要度 | 定義 | 通知方法 | 対応時間 |
|--------|------|---------|---------|
| Sev 0 (Critical) | サービス全停止、データ損失リスク | PagerDuty即時 + 電話 + Slack | 即時 |
| Sev 1 (High) | 主要機能障害、パフォーマンス重大劣化 | PagerDuty + Slack | 15分以内 |
| Sev 2 (Warning) | 一部機能影響、パフォーマンス軽度劣化 | Slack + Email | 1時間以内 |
| Sev 3 (Info) | 情報通知、閾値接近 | Slack | 次営業日 |

### 5.2 アラートルール一覧

#### Sev 0 (Critical) アラート

| アラート名 | 条件 | 通知先 |
|----------|------|--------|
| サービス完全停止 | 全ヘルスチェック失敗（3分間） | ag-critical |
| データベース接続不可 | DB接続エラー100%（2分間） | ag-critical |
| 全レプリカ停止 | Container Appsレプリカ数=0 | ag-critical |
| Front Door全バックエンド異常 | バックエンドヘルス=0% | ag-critical |
| ストレージ容量枯渇 | DB/Redisストレージ > 95% | ag-critical |
| セキュリティ侵害検知 | Defender Critical Alert | ag-critical + ag-security |

#### Sev 1 (High) アラート

| アラート名 | 条件 | 通知先 |
|----------|------|--------|
| HTTP 5xxエラーレート高 | > 5%（5分間平均） | ag-high |
| API応答時間重大劣化 | p95 > 500ms（5分間） | ag-high |
| CPU使用率危険 | > 85%（5分間平均） | ag-high |
| メモリ使用率危険 | > 90%（5分間平均） | ag-high |
| レプリケーション遅延大 | > 30秒 | ag-high |
| Celery Worker全停止 | Worker数=0（2分間） | ag-high |
| フェイルオーバー発生 | Front Doorバックエンド切替検知 | ag-high |

#### Sev 2 (Warning) アラート

| アラート名 | 条件 | 通知先 |
|----------|------|--------|
| HTTP 5xxエラーレート上昇 | > 1%（5分間平均） | ag-warning |
| API応答時間劣化 | p95 > 300ms（5分間） | ag-warning |
| CPU使用率警告 | > 70%（10分間平均） | ag-warning |
| メモリ使用率警告 | > 80%（10分間平均） | ag-warning |
| DB接続数警告 | > 80% of max | ag-warning |
| Redis メモリ警告 | > 70% | ag-warning |
| Celeryキュー滞留 | > 100タスク | ag-warning |
| レプリケーション遅延 | > 5秒 | ag-warning |
| DB ストレージ警告 | > 70% | ag-warning |
| スロークエリ多発 | > 10件/時 | ag-warning |

#### Sev 3 (Info) アラート

| アラート名 | 条件 | 通知先 |
|----------|------|--------|
| スケールアウト発生 | レプリカ数増加 | ag-info |
| スケールイン発生 | レプリカ数減少 | ag-info |
| バックアップ完了 | 定期バックアップ成功 | ag-info |
| 証明書更新完了 | SSL証明書自動更新 | ag-info |
| 新バージョンデプロイ | コンテナイメージ更新検知 | ag-info |

### 5.3 アクショングループ

| グループ名 | 通知方法 | 対象者 |
|----------|---------|--------|
| ag-critical | PagerDuty(P1) + Slack(#ops-critical) + 電話 + Email | 運用L2/L3 + 運用マネージャー |
| ag-high | PagerDuty(P2) + Slack(#ops-itscm) + Email | 運用L2 |
| ag-warning | Slack(#ops-itscm) + Email | 運用L1/L2 |
| ag-info | Slack(#ops-itscm-info) | 運用L1 |
| ag-security | PagerDuty + Slack(#security-alerts) + Email | CISO + セキュリティチーム |

### 5.4 アラート抑制ルール

| ルール | 条件 | 目的 |
|--------|------|------|
| メンテナンスウィンドウ抑制 | 計画メンテナンス中 | 計画作業中のアラート抑制 |
| 重複抑制 | 同一アラート15分以内の重複 | アラート疲れ防止 |
| 依存関係抑制 | 上位障害検知時の下位アラート | 根本原因のみ通知 |
| フラッピング防止 | 3回連続閾値超過で発火 | 一時的な閾値超過の無視 |

## 7. ログ監視

### 7.1 重要ログ監視ルール

```kql
// ルール1: アプリケーションエラー急増検知（5分で10件以上）
ContainerAppConsoleLogs_CL
| where TimeGenerated > ago(5m)
| where Log_s contains "ERROR" or Log_s contains "CRITICAL"
| summarize ErrorCount = count() by ContainerAppName_s
| where ErrorCount > 10

// ルール2: 認証失敗の急増検知（5分で20件以上）
AppRequests
| where TimeGenerated > ago(5m)
| where Url contains "/api/auth/login"
| where ResultCode == "401" or ResultCode == "403"
| summarize FailCount = count()
| where FailCount > 20

// ルール3: DBスロークエリ検知（100ms超）
AppDependencies
| where TimeGenerated > ago(5m)
| where DependencyType == "postgresql"
| where DurationMs > 100
| summarize SlowCount = count(), AvgDuration = avg(DurationMs)
| where SlowCount > 5

// ルール4: 外部依存障害検知
AppDependencies
| where TimeGenerated > ago(5m)
| where Success == false
| summarize FailCount = count() by DependencyType, Target
| where FailCount > 10

// ルール5: 異常なデータアクセスパターン検知
AppRequests
| where TimeGenerated > ago(15m)
| summarize RequestCount = count() by ClientIP = tostring(customDimensions.ClientIP)
| where RequestCount > 1000
```

## 9. 監視運用

### 9.1 監視設定の変更管理

監視設定の変更は変更管理プロセス（ITSCM-RM-005）に従い、以下を遵守する:

- アラート閾値の変更: 標準変更（事前承認済み、±20%以内）
- 新規アラートルールの追加: 通常変更（CAB承認）
- 監視ツールの追加・変更: 通常変更（CAB承認）
- ダッシュボードの変更: 標準変更

### 9.2 定期レビュー

| レビュー項目 | 頻度 | 担当 |
|------------|------|------|
| アラート発火状況レビュー | 週次 | 運用チーム |
| 閾値の適正性確認 | 月次 | 運用+開発 |
| ダッシュボード改善 | 四半期 | 運用チーム |
| 監視ツール評価 | 年次 | アーキテクト+運用 |
| 監視カバレッジ評価 | 四半期 | 運用+セキュリティ |

### 9.3 アラート対応フロー

```
アラート発火
  │
  ├── Sev 0/1: 即時対応
  │    ├── PagerDutyで担当者に通知

  │    ├── 15分以内に初期対応開始

  │
  ├── Sev 2: 確認・対応

  │    ├── 1時間以内に確認

  │
  └── Sev 3: 記録
       ├── Slack #ops-itscm-info に投稿
       └── 次営業日に確認
```
