## 5. STABLE 判定基準

| 変更規模 | 連続成功回数 | 適用例 |
|---------|------------|-------|
| 小規模 | N=2 | コメント修正・軽微な修正 |
| 通常 | N=3 | 機能追加・バグ修正 |
| 重要 | N=5 | 認証・セキュリティ・DB変更 |

**判定条件**: test ✅ + CI ✅ + lint ✅ + build ✅ + error 0 + security critical 0

## 6. ClaudeOS ループ開発サイクル

本プロジェクトは以下のループサイクルで自律開発を実施する。

```
Monitor (1h) → Development (2h) → Verify (2h) → Improvement (3h)
     ↑                                                    |
     └────────────────────── 8h上限 ────────────────────────┘
```

| ループ | 主な作業 | 参照ドキュメント |
|-------|---------|----------------|
| 🔍 Monitor | GitHub状態・CI確認・フェーズ進捗確認 | 本ドキュメント + GitHub Projects #13 |
| 🔨 Development | フェーズ別タスク実装 | 各フェーズ詳細ドキュメント |
| ✔ Verify | テスト・lint・build・セキュリティ確認 | `docs/05_テスト(Testing)/` |
| 🔧 Improvement | カバレッジ向上・技術負債解消 | `docs/04_開発ガイド(Development-Guide)/` |

## 8. リスク管理

| リスク | 影響度 | 発生確率 | 対策 |
|-------|--------|---------|------|
| DB スキーマ変更の影響範囲拡大 | 高 | 中 | Alembic migration + テスト自動化 |
| Azure インフラコスト超過 | 中 | 低 | Container Apps の自動スケール上限設定 |
| セキュリティ脆弱性発見 | 高 | 中 | Dependabot + 週次セキュリティスキャン |
| テストカバレッジ低下 | 中 | 低 | CI でカバレッジ閾値（95%）を強制 |
| フェーズ遅延 | 高 | 中 | 2週毎のバッファ・スコープ削減判断 |
