## 1. フェーズ目標

本番環境相当のステージング環境で統合テストを実施し、実ユーザーによるUAT (User Acceptance Testing) を経て、社内リリース承認を得る。

```
目標:
  - 本番相当ステージング環境での全機能動作確認
  - 実ユーザー（IT 部門・経営企画）による UAT 完了
  - 指摘事項の全件対応
  - 運用引継ぎ準備完了
  - 社内リリース承認取得
```

---

## 2. テスト戦略

### 2.1 統合テスト（Week 23）

| テスト項目 | 内容 | 合格基準 |
|-----------|------|---------|
| バックエンド API 全エンドポイント | ステージング DB 接続での動作確認 | 全 API 正常レスポンス |
| フロントエンド → API 連携 | 実際の HTTP 通信での動作確認 | 全ページエラーなし |
| 認証フロー | Azure AD ログイン → セッション管理 | ログイン/ログアウト正常動作 |
| WebSocket | リアルタイム通知の実動作確認 | 1 秒以内に反映 |
| PDF レポート生成 | 実データでの生成テスト | 正しいフォーマットで生成 |
| メール通知 | ステージング用メールサーバーでの確認 | 通知が正しく届く |
| BIA 計算 | 実際のデータ投入での計算精度確認 | 計算式通りの結果 |
| RTO トラッカー | インシデント発生からの追跡確認 | リアルタイム更新 |

### 2.2 UAT シナリオ（Week 23-24）

#### シナリオ 1: インシデント対応シナリオ
```
想定: 本番 DC 障害が発生した場合の BCP 発動フロー

手順:
  1. 管理者がインシデント登録（severity: p1, scenario: dc_failure）
  2. システムが自動で対象 IT システムの RTO タイマー開始
  3. 復旧手順の確認・実行記録
  4. RTO 超過時のエスカレーション動作確認
  5. インシデントクローズ・事後レポート生成

合格基準:
  - 全ステップが直感的に操作できる
  - 記録が正確に保存される
  - レポートが自動生成される
```

#### シナリオ 2: BCP 計画管理シナリオ
```
想定: 年次 BCP 計画の更新作業

手順:
  1. BIA 評価の新規作成・リスクスコア自動計算
  2. IT システムの RTO/RPO 目標更新
  3. 復旧手順書の改訂
  4. BCP 訓練計画の登録
  5. 準拠状況レポートの出力（ISO20000-1）

合格基準:
  - BIA 計算が業務実態を反映
  - PDF レポートが社内標準フォーマット準拠
```

#### シナリオ 3: 日常監視シナリオ
```
想定: 日常的なシステム状態監視

手順:
  1. ダッシュボードでシステム稼働状況確認
  2. RTO 達成状況のサマリー確認
  3. 過去インシデントの振り返り分析
  4. アクティブインシデントのフィルタリング

合格基準:
  - ダッシュボードが 3 秒以内に表示
  - 必要な情報が一画面で把握できる
```

## 6. 指摘事項管理

| 重大度 | 定義 | 対応期限 |
|-------|------|---------|
| Critical | システムが使用不能・データ損失リスク | 即日対応・即時デプロイ |
| High | 主要機能が正常動作しない | 3 営業日以内 |
| Medium | 機能は動作するが使いにくい | M6 前までに対応 |
| Low | 軽微な UI 改善・誤字等 | 次バージョン以降 |
