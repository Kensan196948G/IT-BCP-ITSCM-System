## 1. 概要

### 1.1 目的

本文書は、IT-BCP-ITSCMシステムに対する全ての変更を体系的に管理し、変更に伴うリスクを最小化するためのプロセスを定義する。ISO20000のサービスマネジメント要件およびISO27001の変更管理要件に準拠する。

### 1.2 適用範囲

本プロセスは以下の変更に適用する:

| 対象 | 例 |
|------|-----|
| アプリケーション変更 | 機能追加、バグ修正、UI変更、API変更 |
| インフラストラクチャ変更 | サーバー構成変更、ネットワーク変更、スケール変更 |
| データベース変更 | スキーマ変更、インデックス変更、データ移行 |
| セキュリティ変更 | アクセス制御変更、暗号化設定変更、証明書更新 |
| 構成変更 | 環境変数変更、設定値変更、機能フラグ変更 |
| ドキュメント変更 | 運用手順書変更、ポリシー変更 |

### 1.3 変更の分類

| 分類 | 定義 | 承認レベル | リードタイム |
|------|------|----------|------------|
| 標準変更 | リスクが低く、手順が確立された定型変更 | 事前承認済み（自動） | 即時〜1営業日 |
| 通常変更 | 計画的な変更、リスク評価が必要 | CAB承認 | 5〜10営業日 |
| 大規模変更 | 影響範囲が広い、複数システムにまたがる変更 | CAB + 経営承認 | 10〜20営業日 |
| 緊急変更 | セキュリティインシデント、サービス障害対応 | 緊急CAB承認 | 即時〜4時間 |

### 1.4 用語定義

| 用語 | 定義 |
|------|------|
| RFC (Request for Change) | 変更要求書 |
| CAB (Change Advisory Board) | 変更諮問委員会 |
| ECAB (Emergency CAB) | 緊急変更諮問委員会 |
| 変更マネージャー | 変更管理プロセスの責任者 |
| 変更オーナー | 個別の変更の実施責任者 |
| PIR (Post Implementation Review) | 変更実施後レビュー |

## 3. 変更影響度評価

### 3.1 影響度評価マトリクス

```
                    影響度
              低      中      高     最大
           ┌──────┬──────┬──────┬──────┐
     高    │  中  │  高  │ 最大 │ 最大 │
           ├──────┼──────┼──────┼──────┤
発   中    │  低  │  中  │  高  │ 最大 │
生         ├──────┼──────┼──────┼──────┤
確   低    │  低  │  低  │  中  │  高  │
率         ├──────┼──────┼──────┼──────┤
     最低  │ 最低 │  低  │  低  │  中  │
           └──────┴──────┴──────┴──────┘

リスクレベル → 承認レベル:
  最低/低: 変更マネージャー承認
  中:     CAB承認
  高:     CAB承認 + テスト強化
  最大:   CAB承認 + 経営承認 + 全テスト
```

### 3.2 影響度の評価基準

| 影響度 | 基準 | 例 |
|--------|------|-----|
| 最大 | サービス全停止リスク、データ損失リスク | DB構造の破壊的変更、ネットワーク構成の大幅変更 |
| 高 | 主要機能への影響、パフォーマンス劣化リスク | 認証システム変更、APIスキーマ変更 |
| 中 | 一部機能への影響、限定的なユーザー影響 | UI変更、新機能追加、設定値変更 |
| 低 | サービスへの影響なし | ドキュメント更新、ログ設定変更 |

### 3.3 発生確率の評価基準

| 発生確率 | 基準 | 根拠 |
|---------|------|------|
| 高 | 過去に類似変更で問題が発生した | 過去のインシデント記録 |
| 中 | 初めての種類の変更、テスト不十分 | テストカバレッジ、経験 |
| 低 | 十分なテスト済み、確立された手順 | テスト結果、過去の成功実績 |
| 最低 | 定型的な変更、自動化されている | 標準変更の実績 |

### 3.4 技術的影響度評価チェックリスト

```markdown

### APIへの影響
```

## 4. 変更承認フロー

### 4.1 CAB（変更諮問委員会）

#### 構成メンバー

| 役割 | メンバー | 責務 |
|------|---------|------|
| 議長（変更マネージャー） | プロジェクトマネージャー | 会議進行、最終判定 |
| テクニカルアドバイザー | テックリード / アーキテクト | 技術的評価 |
| QA代表 | QAリーダー | テスト観点での評価 |
| セキュリティ代表 | セキュリティエンジニア | セキュリティ観点での評価 |
| 運用代表 | 運用リーダー | 運用影響の評価 |
| ビジネス代表 | プロダクトオーナー | ビジネス影響の評価 |

#### CAB開催スケジュール

| 種別 | 頻度 | 時間 | 対象 |
|------|------|------|------|
| 定例CAB | 毎週水曜日 10:00-11:00 | 1時間 | 通常変更 |
| 臨時CAB | 必要に応じて | 30分〜1時間 | 大規模変更 |
| ECAB | 緊急時即時 | 15分〜30分 | 緊急変更 |

#### CAB審議プロセス

```
1. 変更要求の提示（変更オーナー: 5分）
   - 変更内容の説明
   - 影響度評価結果の報告
   - テスト計画の説明

2. 質疑応答（全メンバー: 10分）
   - 技術的な質問
   - リスクに関する質問
   - 運用影響に関する質問

3. リスク評価（全メンバー: 5分）
   - リスクレベルの合意
   - 追加の軽減策の検討

4. 判定（議長: 5分）
   - 承認 / 条件付き承認 / 保留 / 却下
   - 条件・コメントの記録
```

### 4.2 承認判定基準

| 判定 | 条件 |
|------|------|
| 承認 | リスクが許容範囲内、テスト計画が適切、ロールバック計画が準備済み |
| 条件付き承認 | 追加テスト、追加レビュー、ドキュメント修正等の条件を満たした上で承認 |
| 保留 | 情報不足、追加の影響分析が必要 |
| 却下 | リスクが許容範囲外、代替案の検討が必要 |

### 4.3 承認権限マトリクス

| 変更分類 | 変更マネージャー | CAB | ECAB | CTO | CISO |
|---------|---------------|-----|------|-----|------|
| 標準変更 | 事前承認済み | - | - | - | - |
| 通常変更（低リスク） | 必須 | 報告 | - | - | - |
| 通常変更（中リスク） | 必須 | 必須 | - | - | - |
| 通常変更（高リスク） | 必須 | 必須 | - | 必須 | - |
| 大規模変更 | 必須 | 必須 | - | 必須 | 必須 |
| 緊急変更 | 必須 | - | 必須 | 通知 | 通知 |
| セキュリティ関連変更 | 必須 | 必須 | - | - | 必須 |

## 5. 緊急変更プロセス

### 5.1 緊急変更の定義

以下の条件に該当する場合、緊急変更プロセスを適用する:

| 条件 | 例 |
|------|-----|
| サービス停止の復旧 | 本番環境の障害復旧、ホットフィックス |
| セキュリティインシデント対応 | 脆弱性の緊急修正、不正アクセス対応 |
| データ損失防止 | データ破損の修復、バックアップリストア |
| 法規制・コンプライアンス対応 | 期限付きの規制対応 |

### 5.2 緊急変更フロー

```
インシデント検知
  │
  ├── 1. 初期評価（5分以内）
  │    ├── インシデントの重大度判定
  │    ├── 緊急変更の必要性判断
  │    └── 変更オーナーのアサイン
  │
  ├── 2. ECAB召集（15分以内）
  │    ├── 最低3名のECABメンバー参加
  │    ├── 変更内容の説明
  │    ├── リスク評価（簡易）
  │    └── 承認判定
  │
  ├── 3. 変更実施
  │    ├── 修正の実装
  │    ├── 最低限のテスト（クリティカルパスのみ）
  │    ├── デプロイ（緊急手順）
  │    └── 動作確認
  │
  ├── 4. 事後対応（24時間以内）
  │    ├── RFC の事後登録
  │    ├── 詳細な影響分析
  │    ├── 追加テストの実施
  │    └── ドキュメント更新
  │
  └── 5. PIR（5営業日以内）
       ├── ポストモーテム
       ├── 根本原因分析
       ├── 再発防止策
       └── プロセス改善
```

### 5.3 ECAB（緊急変更諮問委員会）

| 項目 | 内容 |
|------|------|
| 最低参加者数 | 3名（変更マネージャー + テクニカルアドバイザー + 運用代表） |
| 召集方法 | 電話/Slack/Teams即時連絡 |
| 審議時間 | 最大15分 |
| 判定方法 | 多数決（最低2名の承認） |
| 営業時間外 | オンコール担当者による判断 |

## 7. 変更管理の改善

### 7.1 PIR（変更後レビュー）

全ての通常変更・大規模変更・緊急変更について、変更実施後にPIRを実施する。

#### PIRテンプレート

```markdown
## PIR（Post Implementation Review）

### 基本情報
- RFC番号: RFC-YYYY-NNNN
- 変更内容:
- 実施日: YYYY-MM-DD
- レビュー日: YYYY-MM-DD

### 評価
| 評価項目 | 結果 | コメント |
|---------|------|---------|
| 変更は計画通り実施されたか | はい/いいえ | |
| 目的は達成されたか | はい/いいえ | |
| 予期しない影響はあったか | はい/いいえ | |
| ロールバックは必要だったか | はい/いいえ | |
| 所要時間は計画通りか | はい/いいえ | |

### 教訓
（得られた教訓）

### 改善提案
| 改善項目 | 担当 | 期限 |
|---------|------|------|
| | | |
```

### 7.2 プロセス改善サイクル

```
  ┌──────────┐
  │  Plan    │ ← 改善計画策定（四半期ごと）
  └────┬─────┘
       │
  ┌────▼─────┐
  │   Do     │ ← 改善策実施
  └────┬─────┘
       │
  ┌────▼─────┐
  │  Check   │ ← KPIモニタリング、PIRレビュー
  └────┬─────┘
       │
  ┌────▼─────┐
  │   Act    │ ← 改善結果の反映、プロセス更新
  └──────────┘
```

### 7.3 定期レビュー

| レビュー | 頻度 | 参加者 | 内容 |
|---------|------|--------|------|
| 週次CABレビュー | 毎週 | CABメンバー | 今週の変更レビュー、来週の変更計画 |
| 月次変更管理レビュー | 毎月 | 変更マネージャー + CTO | KPIレビュー、傾向分析 |
| 四半期プロセスレビュー | 四半期 | 全ステークホルダー | プロセス改善、ポリシー見直し |
| 年次監査 | 年次 | 外部監査 + 全ステークホルダー | ISO20000/27001準拠確認 |
