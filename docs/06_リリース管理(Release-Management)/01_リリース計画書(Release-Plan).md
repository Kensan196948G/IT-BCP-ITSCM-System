## 1. 概要

### 1.1 目的

本文書は、IT事業継続管理システム（IT-BCP-ITSCM-System）のリリース計画を定義する。災害・サイバー攻撃時のIT復旧計画・BCP訓練・RTOダッシュボード統合プラットフォームとして、フェーズごとの機能リリースを安全かつ確実に実施するための計画を策定する。

### 1.2 適用範囲

- フロントエンド: Next.js 14 / TypeScript / PWA対応
- バックエンド: Python FastAPI / PostgreSQL（Geo冗長）/ Redis Cluster / Celery Worker
- インフラ: Azure Container Apps マルチリージョン（East Japan Primary + West Japan Standby）
- CDN/ロードバランサ: Azure Front Door
- CI/CD: GitHub Actions

### 1.3 リリースポリシー

| ポリシー項目 | 内容 |
|------------|------|
| リリース頻度 | 定期リリース: 月1回（第2火曜日）、緊急リリース: 随時 |
| リリースウィンドウ | 通常: 火曜日 22:00-02:00 JST、緊急: 承認後即時 |
| コードフリーズ | リリース日の5営業日前 |
| リリース禁止期間 | 年末年始（12/28-1/4）、GW、大型連休期間 |
| ロールバック猶予 | リリース後72時間以内はロールバック即時可能 |

## 3. リリースバージョン管理

### 3.1 セマンティックバージョニング規則

本プロジェクトでは、Semantic Versioning 2.0.0（SemVer）を採用する。

```
MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]

例: 1.2.3-beta.1+build.456
```

| 要素 | 意味 | インクリメント条件 |
|------|------|-----------------|
| MAJOR | メジャー | 後方互換性のないAPI変更 |
| MINOR | マイナー | 後方互換性を保った機能追加 |
| PATCH | パッチ | 後方互換性を保ったバグ修正 |
| PRERELEASE | プレリリース | alpha → beta → rc の順で遷移 |
| BUILD | ビルド | CI/CDでの自動付番（Git SHA等） |

### 3.2 プレリリースバージョン体系

```
alpha.N  → 開発中、機能未完成、内部テスト用
beta.N   → 機能完成、外部テスト可能
rc.N     → リリース候補、最終確認段階
(なし)    → GA（正式リリース）
```

### 3.3 バージョン命名規則

| 種別 | 形式 | 例 | 用途 |
|------|------|-----|------|
| フェーズリリース | vMAJOR.MINOR.0 | v1.1.0 | 各フェーズの正式リリース |
| スプリントリリース | vMAJOR.MINOR.0-pre.N | v1.1.0-alpha.2 | スプリント単位の中間リリース |
| ホットフィックス | vMAJOR.MINOR.PATCH | v1.0.1 | 緊急バグ修正 |
| セキュリティパッチ | vMAJOR.MINOR.PATCH | v1.0.2 | セキュリティ脆弱性修正 |

### 3.4 ブランチ戦略とバージョン対応

```
main ─────────────────────────────────────────────────────→ 本番環境
  │                                                          (v1.0.0, v1.0.1, ...)
  ├── release/v1.0.0 ──→ タグ: v1.0.0 ──→ マージ → main
  ├── release/v1.1.0 ──→ タグ: v1.1.0 ──→ マージ → main
  │
develop ──────────────────────────────────────────────────→ 開発統合
  ├── feature/ITSCM-001-auth ──→ マージ → develop
  ├── feature/ITSCM-002-dashboard ──→ マージ → develop
  ├── hotfix/v1.0.1 ──→ マージ → main + develop
  └── bugfix/ITSCM-100-fix ──→ マージ → develop
```

### 3.5 Gitタグ管理

```bash
# リリースタグの作成
git tag -a v1.0.0 -m "Phase 1 GA Release - IT-BCP-ITSCM-System v1.0.0"

# タグの署名（GPG）
git tag -s v1.0.0 -m "Phase 1 GA Release - IT-BCP-ITSCM-System v1.0.0"

# タグのプッシュ
git push origin v1.0.0
```

### 3.6 コンテナイメージのタグ戦略

| タグ形式 | 用途 | 例 |
|---------|------|-----|
| `vMAJOR.MINOR.PATCH` | リリースバージョン | `itscm-api:v1.0.0` |
| `vMAJOR.MINOR.PATCH-PRERELEASE` | プレリリース | `itscm-api:v1.0.0-rc.1` |
| `sha-XXXXXXX` | コミットSHA | `itscm-api:sha-abc1234` |
| `latest` | 最新安定版 | `itscm-api:latest` |
| `develop` | 開発最新 | `itscm-api:develop` |

| DNS切替 | Azure Front Door設定確認 | ヘルスチェック連動切替正常 |

### 6.3 検証スケジュール

```
コードフリーズ           ステージングデプロイ       検証期間              判定会議
    │                      │                    │                   │
    ├── Day 0 ──────────── Day 1 ──────────── Day 2-4 ──────────── Day 5 ──→ リリース
    │                      │                    │                   │
    │                      ├ 自動デプロイ         ├ 機能検証(Day 2)     ├ 判定会議
    │                      ├ スモークテスト        ├ 性能検証(Day 3)     ├ Go/No-Go決定
    │                      └ 基本動作確認         ├ セキュリティ(Day 3)  └ リリース承認
    │                                           └ インフラ検証(Day 4)
```

## 7. リリース承認フロー

### 7.1 承認フロー全体像

```
開発チーム → QAチーム → セキュリティチーム → 運用チーム → 経営層
   │            │             │               │           │
   ├ コード品質   ├ テスト結果    ├ セキュリティ     ├ 運用準備     ├ ビジネスリスク
   ├ テスト結果   ├ バグ状況     ├ 審査結果       ├ 監視準備     ├ コスト確認
   └ 技術レビュー  └ パフォーマンス └ コンプライアンス  └ ロールバック  └ 最終承認
                   結果                        計画確認
```

### 7.2 各承認ステージの詳細

#### ステージ1: 開発チーム承認

| 確認項目 | 確認者 | 承認基準 |
|---------|--------|---------|
| コードレビュー完了 | テックリード | 全PRレビュー・承認済み |
| 単体テスト合格 | 開発リーダー | カバレッジ85%以上、全PASS |
| 静的解析クリア | テックリード | Critical/High 0件 |
| ビルド成功 | CI/CD担当 | 全パイプラインGREEN |
| 技術的負債評価 | アーキテクト | 許容範囲内 |
| リリースノート作成 | 開発リーダー | 変更内容記載完了 |

**承認者**: 開発マネージャー
**承認期限**: コードフリーズ後1営業日以内

#### ステージ2: QAチーム承認

| 確認項目 | 確認者 | 承認基準 |
|---------|--------|---------|
| 結合テスト結果 | QAリーダー | 全テストPASS |
| E2Eテスト結果 | QAリーダー | 全シナリオPASS |
| パフォーマンステスト | 性能テスト担当 | 全目標値達成 |
| 回帰テスト結果 | QAチーム | 既存機能影響なし |
| UAT結果 | ビジネスオーナー | 受入基準充足 |
| バグ残件確認 | QAリーダー | Critical 0件、High 0件 |

**承認者**: QAマネージャー
**承認期限**: ステージング検証完了後1営業日以内

#### ステージ3: セキュリティチーム承認

| 確認項目 | 確認者 | 承認基準 |
|---------|--------|---------|
| 脆弱性スキャン結果 | セキュリティエンジニア | Critical/High 0件 |
| ペネトレーションテスト | 外部セキュリティ監査 | 重大脆弱性0件 |
| コンプライアンス確認 | コンプライアンス担当 | ISO27001要件充足 |
| データ保護確認 | DPO | 個人情報保護法準拠 |
| アクセス制御確認 | セキュリティチーム | RBAC正常動作 |
| 暗号化確認 | セキュリティエンジニア | 通信・保存データ暗号化確認 |

**承認者**: CISO（最高情報セキュリティ責任者）
**承認期限**: セキュリティ検証完了後2営業日以内

#### ステージ4: 運用チーム承認

| 確認項目 | 確認者 | 承認基準 |
|---------|--------|---------|
| 運用手順書確認 | 運用リーダー | 手順書最新化完了 |
| 監視設定確認 | 監視担当 | 全監視項目設定完了 |
| アラート設定確認 | 監視担当 | アラート疎通確認済み |
| ロールバック手順確認 | 運用リーダー | ロールバック手順テスト済み |
| バックアップ確認 | DBA | バックアップ取得確認 |
| 障害対応手順確認 | 運用チーム | 障害対応手順レビュー済み |
| DR手順確認 | 運用リーダー | フェイルオーバーテスト済み |

**承認者**: 運用マネージャー
**承認期限**: リリース日前日まで

#### ステージ5: 経営承認

| 確認項目 | 確認者 | 承認基準 |
|---------|--------|---------|
| ビジネスリスク評価 | CTO | リスク許容範囲内 |
| コスト確認 | 経営企画 | 予算内 |
| スケジュール確認 | PMO | 計画通り |
| ステークホルダー合意 | プロジェクトマネージャー | 関係者合意取得 |
| Go/No-Go判定 | CTO/CIO | 全承認ステージクリア |

**承認者**: CTO / CIO
**承認期限**: リリース日前日まで

### 7.3 緊急リリースの承認フロー

セキュリティインシデントやシステム障害に対する緊急リリースの場合:

```
報告 → 緊急変更要求 → セキュリティ/運用 → CTO承認 → 緊急リリース
                       （同時並行審査）
```

| ステップ | 所要時間 | 承認者 |
|---------|---------|--------|
| 緊急変更要求作成 | 30分以内 | 開発リーダー |
| セキュリティ・運用同時審査 | 1時間以内 | CISO + 運用マネージャー |
| CTO承認 | 30分以内 | CTO |
| リリース実行 | リリース手順に準拠 | リリース担当 |
| 事後レビュー | リリース後3営業日以内 | 全ステークホルダー |

### 7.4 承認記録管理

全リリース承認は以下の形式で記録・保管する:

| 記録項目 | 内容 |
|---------|------|
| リリースバージョン | vX.Y.Z |
| 承認日時 | YYYY-MM-DD HH:MM:SS |
| 承認者 | 氏名・役職 |
| 承認ステージ | ステージ1〜5 |
| 承認結果 | 承認/条件付き承認/却下 |
| 条件・コメント | 条件付き承認の場合の条件等 |
| エビデンス | テスト結果URL、スキャンレポート等 |

保管場所: GitHub Releases + 社内承認管理システム
保管期間: 5年以上（ISO27001要件）

## 8. リスク管理

### 8.1 リリースに関するリスク一覧

| リスクID | リスク | 影響度 | 発生確率 | 対策 |
|---------|--------|--------|---------|------|
| RR-001 | ステージングと本番の環境差異による障害 | 高 | 中 | Infrastructure as Code徹底、環境パリティ確保 |
| RR-002 | DBマイグレーション失敗 | 高 | 低 | マイグレーションテスト、ロールバックスクリプト準備 |
| RR-003 | マルチリージョンデプロイ時の不整合 | 高 | 中 | Blue-Greenデプロイ、段階的ロールアウト |
| RR-004 | 依存サービスの障害 | 中 | 中 | ヘルスチェック、サーキットブレーカー |
| RR-005 | リリース中のセキュリティインシデント | 高 | 低 | リリース中止基準、緊急対応手順 |
| RR-006 | ロールバック不可能な状態 | 高 | 低 | Forward-fixプレイブック、データバックアップ |

### 8.2 リリース中止基準

以下の条件に該当する場合、リリースを中止する:

- ステージング検証で重大バグ（Critical）が発見された場合
- セキュリティスキャンでHigh以上の脆弱性が検出された場合
- パフォーマンステストで目標値を20%以上下回る場合
- フェイルオーバーテストが失敗した場合
- 承認フローの必須承認が得られない場合
- リリースウィンドウ内にデプロイが完了しない見込みの場合
