#!/bin/bash # 使用例: project-sync.sh ISSUE_ID Development ISSUE_ID=$1 STATUS=$2 PROJECT_ID="YOUR_PROJECT_ID" FIELD_ID="STATUS_FIELD_ID" gh api graphql -f query=' mutation($project:ID!, $item:ID!, $field:ID!, $value:String!) { updateProjectV2ItemFieldValue(input:{ projectId:$project itemId:$item fieldId:$field value:{singleSelectOptionId:$value} }) { projectV2Item { id } } }' \ -f project="$PROJECT_ID" \ -f item="$ISSUE_ID" \ -f field="$FIELD_ID" \ -f value="$STATUS"

#!/bin/bash # 自動トリガー: TODO/FIXME/CI失敗/テスト不足 TITLE="$1" BODY="$2" gh issue create \ --title "$TITLE" \ --body "$BODY" \ --label "auto-generated" # 自動トリガー条件: # - TODO検出 # - FIXME検出 # - CI失敗 # - テスト不足

ベストプラクティス 8原則（仕様書準拠）

行動原則：Small change / Test everything / Stable first / Deploy safely / Improve continuously / Stop at 8h safely

1

/loop 30m Monitor — 09:00に正確発火・30分で完結させる

仕様書明記：「アイドルなら正確に発火」

Monitor Loopは30分で完結させる設計です。30分を超えるとDevelopment Loopの開始時刻が後ろにずれ、17:00のLoopGuard停止までに全Loopが完了しなくなります。

  • GitHub Project確認・Issue優先度ソートは最大15分以内
  • Dev Factoryのタスク自動生成（最大3件）は自動実行 — 人間は確認のみ
  • 前日Improvement Loopの残分（2h）もMonitor冒頭に継続確認
  • 30分を超えたら強制的にDevelopmentに移行する（Monitor延長禁止）
2

GitHub Project を唯一の司令盤として徹底活用

仕様書明記：「GitHub Projectsを中心にタスク状態を管理する」

ClaudeOS v4はGitHub Projectと完全連動して動作します。scripts/project-sync.sh が全ステータス変更を自動実行します。

  • Boot Sequence時に全Issue状態を確認 → Inbox→Backlog→Ready→Design→Development に更新
  • scripts/project-sync.sh でStatus変更を全て自動化（graphql API使用）
  • 8時間終了時は必ず実態と整合するStatusに更新（Blocked or Backlogへ）
  • Issue未登録のままLoop開始は「Scrum MasterとDev Factoryが機能しない」原因
3

Small change 原則 — 頻繁commitでAuto Repairを高速化

仕様書明記：「Small change」行動原則

1回のcommitを小さく保つことで、CI失敗時のAuto Repair（最大15回）が正確な失敗箇所を特定でき、修復速度が大幅に上がります。

  • 1commit = 1機能・1修正。300行以上のcommitは禁止
  • WorkTree: feature/[曜日]-[task] で1タスク1ブランチを厳守
  • TODO/FIXMEを検出したら scripts/create-issue.sh で即座にIssue化
  • main直接push → LoopGuardが検知→自動阻止。PR経由のみ許可
4

STABLE判定はN=3（デフォルト）を厳守

仕様書明記：「デフォルトN=3 / 小規模N=2 / 重要N=5」

STABLEと判定されてはじめて commit→push→PR→merge→deploy の完全パイプラインが実行されます。N=3は「3回連続成功」の意味です。

  • 通常変更（既存統合タスク）→ N=3（デフォルト）
  • 小規模変更（1ファイル修正等）→ N=2
  • 重要変更（Security・認証・secrets変更）→ N=5。自動承認対象外
  • STABLE未達 + 8時間到達 → STABLEを待たずLoopGuardが安全停止
5

Improvement(/loop 4h)は「2h実動 + 2h翌日継続」設計で運用

17:00 LoopGuard停止のため、残り2時間は翌日のMonitorで継続

Improvement Loopは4時間設計ですが、15:00〜17:00の2時間しか実動しません。残り2時間は翌日のMonitor(/loop 30m)冒頭に「前日Improvement継続」として発火します。

  • 15:00〜17:00（2h）：パターン学習・ループ最適化を集中実施
  • 翌日09:00 Monitor冒頭：前日Improvement残分（2h）を優先処理
  • STABLE達成後の self-evolution.md（Evolutionトリガー）もここで実行
  • 金曜17:00停止分のImprovement残は翌週月曜に持越し
6

Agent Teamsの会話を必ず可視化する

仕様書明記：「Agent Teamsの会話は必ず可視化してください」

各AgentがどのLoopでどんな判断をしたかを可視化することで、8時間終了時の最終報告（15項目）が自動生成されます。

  • CTO：優先順位判断・8時間終了時の最終判断 → 可視化必須
  • Security：PR時のセキュリティレビュー結果 → ログに残す
  • Architect：architecture-check-loop.mdの審査結果 → PRコメントとして記録
  • 会話の可視化なしで最終報告を生成しようとすると「根拠なし報告」になる
7

最終報告は16:50から開始 — 9項目 + 8h終了時追加6項目

仕様書明記：「終了時に以下を報告してください」（15項目）

16:50から最終報告生成を開始することで、17:00のLoopGuard強制終了前に全15項目が完成します。この報告書が翌日のBoot Sequenceの材料になります。

  • 基本9項目：開発内容・修正内容・テスト結果・CI結果・Project更新・PR/merge・deploy結果・残課題・次フェーズ提案
  • 8h終了時追加6項目：開始時刻・終了時刻・総作業時間・未完了タスク・再開ポイント・継続優先順位
  • この最終報告書 = 翌日のCLAUDE.md（タスク指示書）の材料
  • 16:50時点でImprovement実行中でも最終報告を優先して開始する
8

LoopGuardを「安全装置」として信頼する — 人間は止めなくてよい

仕様書明記：「Loop Guardが最優先」「Stop at 8 hours safely」

LoopGuardは4つの停止条件を自動監視します。人間が手動で止める必要はありません。むしろLoopGuardが止まった場合は「根本的な問題がある」サインです。

  • 8時間（17:00）到達 → 安全終了 + 最終報告（AI CTO実行）
  • 同一エラー3回 → 停止 + 人間確認が必要なサイン
  • CI失敗5回連続 → 停止（Auto Repair最大15回はLoopGuard管理下）
  • Securityクリティカル問題 → 即時停止 → secrets/認証/権限変更の手動確認が必須
