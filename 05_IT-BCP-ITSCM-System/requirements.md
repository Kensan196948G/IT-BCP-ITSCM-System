## 1. 目的・背景

### 1.1 背景

建設業は社会インフラを担う重要産業であり、大規模地震・台風・感染症・サイバー攻撃等の災害・インシデント発生時においても、工事の進捗管理・安全管理・契約管理を継続する必要がある。

みらい建設工業では、BCP（Business Continuity Plan）は策定されているものの、以下の課題が存在する。

- IT視点での事業継続計画（ITSCM: IT Service Continuity Management）が未整備
- RTO（目標復旧時間）・RPO（目標復旧時点）が各システムで未定義
- BCP訓練がデスクトップ演習にとどまり、実際のIT復旧手順が未検証
- 災害時のIT代替手段（フォールバック）が明文化されていない
- 現場作業員への緊急連絡・情報提供手段が不十分

### 1.2 目的

IT視点での事業継続計画を策定・管理・訓練するためのシステム基盤を構築し、いかなる災害・サイバー攻撃においても基幹IT機能を迅速に復旧できる体制を整備する。

## 3. スコープ

### 3.1 対象シナリオ

| シナリオ | 想定規模 | 優先度 |
|---------|---------|--------|
| 大規模地震（首都直下型・南海トラフ） | 全社 | 最高 |
| サイバー攻撃（ランサムウェア） | 全社ITシステム | 最高 |
| 主要DC（データセンタ）障害 | オンプレミスシステム全体 | 高 |
| クラウドサービス広域障害（Azure） | クラウドサービス全体 | 高 |
| 感染症拡大・パンデミック | 人員不足・リモート対応 | 高 |
| 重要サプライヤー（IT事業者）障害 | 対象サービス | 中 |
| 局所的システム障害 | 対象システム | 中 |

### 3.2 対象システム・RTO/RPO定義

| システム | RTO（目標復旧時間） | RPO（目標復旧時点） | 優先度 |
|---------|----------------|----------------|--------|
| Active Directory | 4時間 | 1時間 | 最高 |
| Entra ID / Microsoft 365 | クラウドSLA依存 | クラウドSLA依存 | 最高 |
| Exchange Online | クラウドSLA依存 | クラウドSLA依存 | 高 |
| ファイルサーバ | 8時間 | 24時間 | 高 |
| DeskNet's Neo | 24時間 | 24時間 | 中 |
| AppSuite | 48時間 | 24時間 | 中 |
| ITSM-System | 8時間 | 4時間 | 高 |
| SIEM Platform | 8時間 | 4時間 | 高 |

## 5. 非機能要件

### 5.1 可用性要件（重要）

本システム自体が災害時に利用されるため、高可用性が必須。

| 項目 | 要件値 |
|------|--------|
| 稼働率 | 99.9%以上 |
| 災害時可用性 | **オフライン動作対応**（ネットワーク断絶時も使用可能） |
| 冗長構成 | マルチリージョン（東日本 + 西日本） |
| RTO（本システム自体） | **15分以内**（他システム復旧に先立って復旧） |

### 5.2 アクセス要件

| 項目 | 要件値 |
|------|--------|
| オフライン動作 | PWA対応によるオフライン閲覧 |
| モバイル対応 | iOS/Android スマートフォン対応 |
| 外部アクセス | 社外からのHTTPS接続対応 |
